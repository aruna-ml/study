Machine Learning Foundations Regression & Classification

Master the core ML model families, loss functions, and evaluation metrics essential for building robust cloud capacity planning systems. This section bridges theoretical foundations with practical iCloud applications.

1.1 Core Model Families

Linear Regression & Regularization

Linear regression forms the foundation of many cloud resource forecasting models. For a dataset with n samples, we seek to minimize:

OLS Objective: $$\text{minimize } \frac{1}{2n} \sum_{i=1}^{n}(y_i - \mathbf{w}^T\mathbf{x}_i)^2$$

Ridge Regression (L2 Regularization): Adds a penalty proportional to the square of coefficients:

Ridge Loss: $$J(\mathbf{w}) = \frac{1}{2n}\sum_{i=1}^{n}(y_i - \mathbf{w}^T\mathbf{x}_i)^2 + \lambda\|\mathbf{w}\|_2^2$$

Ridge works well when you have many correlated features (e.g., CPU, memory, network utilization are correlated). It shrinks all coefficients slightly but keeps all features. Closed-form solution exists: \(\mathbf{w} = (X^TX + 2\lambda I)^{-1}X^Ty\)

Lasso Regression (L1 Regularization): Adds a penalty proportional to the absolute value of coefficients:

Lasso Loss: $$J(\mathbf{w}) = \frac{1}{2n}\sum_{i=1}^{n}(y_i - \mathbf{w}^T\mathbf{x}_i)^2 + \lambda\|\mathbf{w}\|_1$$

Lasso performs feature selection by driving less important coefficients exactly to zero. Use when you suspect only a few features truly drive capacity (e.g., maybe only peak request rate and time-of-day matter). No closed-form; requires iterative solvers like coordinate descent.

Cloud Application: Cost Modeling

Ridge regression excels at modeling data center cooling costs as a linear function of power draw, ambient temperature, and utilization rate—all naturally correlated. If you suspect only peak utilization truly drives procurement decisions, use Lasso to identify the most predictive features and eliminate noise.

Logistic Regression

For binary classification, logistic regression models the probability of class 1:

Logistic Function: $$P(y=1|\mathbf{x}) = \sigma(\mathbf{w}^T\mathbf{x}) = \frac{1}{1+e^{-\mathbf{w}^T\mathbf{x}}}$$

The decision boundary occurs at \(\mathbf{w}^T\mathbf{x} = 0\). Training uses maximum likelihood estimation, which for binary classification is equivalent to minimizing log loss (cross-entropy). This is globally convex, so gradient descent finds the optimal solution reliably.

iCloud Classification Task

Classify servers as over-provisioned (label 1) vs appropriately provisioned (label 0) based on historical utilization. A logistic regression provides interpretable coefficients: positive coeff on "peak_util_ratio" means high peak usage increases outage probability.

Gradient Boosted Trees (XGBoost, LightGBM, CatBoost)

Boosting sequentially trains weak learners (usually shallow trees), each correcting errors of previous trees. The ensemble prediction is:

Boosting Ensemble: $$\hat{y}_i = \sum_{m=1}^{M} f_m(\mathbf{x}_i)$$

where each \(f_m\) is a decision tree. Key hyperparameters:

  • Learning rate (shrinkage): Controls step size; lower learning rates (0.01–0.1) reduce overfitting
  • Tree depth: Typically 3–8 for tabular data; deeper trees capture interactions but risk overfitting
  • Subsampling: Each tree trains on random subset of rows/columns; acts as regularization

Why Gradient Boosting Dominates: Cloud telemetry is high-dimensional and noisy (CPU, memory, network, disk I/O, request rate, time features, etc.). Trees naturally capture nonlinear relationships and feature interactions without manual engineering. XGBoost/LightGBM are fast, handle missing values, and provide feature importance scores.

Tabular Cloud Data

iCloud capacity forecasts typically use 50+ features (hourly lags of CPU/memory/network, rolling 24h/7d stats, calendar features, external events). Gradient boosting beats linear models and shallow neural networks because it automatically learns which feature combinations matter.

Random Forests

Bagging (Bootstrap Aggregating) trains multiple independent models on random subsets of data and averages predictions. Random Forests apply bagging to decision trees, additionally randomizing which features each tree considers at each split:

Bagging Estimator: $$\hat{f}(x) = \frac{1}{B}\sum_{b=1}^{B} f_b(x)$$

Out-of-Bag (OOB) Error: When training tree b, roughly 1/3 of samples are not selected (bootstrap sampling with replacement). These can serve as a free validation set, providing an unbiased error estimate without need for separate validation data.

Feature Importance: Computed as total decrease in impurity (Gini/entropy) across all trees for each feature. Useful for understanding which cloud metrics drive forecasts, but can be biased toward high-cardinality features.

Random Forests vs Gradient Boosting

Random Forests reduce variance through parallelization (bagging). Gradient Boosting sequentially reduces bias. For capacity forecasting, gradient boosting typically outperforms Random Forests (lower RMSE), but Random Forests are easier to parallelize and less prone to overfitting with fewer hyperparameters to tune.

Neural Networks (Multi-Layer Perceptrons)

A feedforward neural network with hidden layers can approximate any continuous function (universal approximation theorem). For a network with one hidden layer:

MLP Forward Pass: $$\hat{y} = W_2 \sigma(W_1 \mathbf{x} + b_1) + b_2$$

where \(\sigma\) is an activation function (ReLU, tanh, sigmoid). Training via backpropagation computes gradients of all parameters via chain rule, enabling stochastic gradient descent.

When to Use: Neural networks excel when:

  • You have very large datasets (millions of samples)
  • Complex, highly nonlinear feature interactions exist
  • Raw input representations benefit from learned feature hierarchies

When NOT to Use: Neural networks struggle with:

  • Small datasets (< 10k samples) prone to overfitting
  • Interpretability requirements (black-box)
  • Categorical features with high cardinality (inefficient encoding)
  • Missing values (require imputation preprocessing)

For cloud capacity forecasting with ~100k samples of daily/hourly aggregates, gradient boosting typically outperforms MLPs because the feature engineering is already done and the relationship is not so complex as to require learned hierarchies.

1.2 Loss Functions

Mean Squared Error (MSE)

MSE Formula: $$\text{MSE} = \frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2$$

MSE penalizes larger errors quadratically. A forecast that's off by 2x gets 4x penalty. Properties:

  • Differentiable everywhere (good for gradient-based optimization)
  • Sensitive to outliers (one huge error dominates the loss)
  • Minimizer is the conditional mean: \(\arg\min_{\hat{y}} E[(y - \hat{y})^2] = E[y|x]\)

Cloud Application: MSE works well if you believe forecast errors are normally distributed and outliers represent true anomalies (e.g., a sudden traffic surge due to viral content). But if your data has genuine extreme values (demand spikes during key announcements), MSE will overfit to capture them.

Mean Absolute Error (MAE)

MAE Formula: $$\text{MAE} = \frac{1}{n}\sum_{i=1}^{n}|y_i - \hat{y}_i|$$

MAE penalizes errors linearly. Properties:

  • Robust to outliers (a huge error only adds linearly)
  • Non-differentiable at zero (can cause issues with some optimizers)
  • Minimizer is the conditional median: \(\arg\min_{\hat{y}} E[|y - \hat{y}|] = \text{median}(y|x)\)
  • Directly interpretable: average absolute deviation from forecast

Cloud Application: If your capacity data has true outliers (e.g., once a month a data center goes into maintenance and demand spikes to nearby regions), MAE is more robust. Optimizing MAE will produce median forecasts, not mean forecasts—appropriate if you care about typical behavior, not worst-case.

Log Loss / Binary Cross-Entropy

Log Loss Formula: $$\text{Log Loss} = -\frac{1}{n}\sum_{i=1}^{n}[y_i \log(p_i) + (1-y_i)\log(1-p_i)]$$

where \(y_i \in \{0, 1\}\) is the true label and \(p_i = P(y_i=1|\mathbf{x}_i) \in (0,1)\) is the predicted probability.

Information-Theoretic Interpretation: Log loss is the Kullback-Leibler divergence between true label distribution (point mass at 0 or 1) and predicted distribution (Bernoulli with parameter \(p\)). It heavily penalizes confident wrong predictions. If you predict \(p=0.9\) but the true label is 0, the loss is \(-\log(0.1) \approx 2.3\)—much worse than being uncertain (\(p=0.5\) gives loss \(\approx 0.69\)).

Cloud Classification: Use log loss when training models to classify servers as over-provisioned. It naturally produces well-calibrated probabilities: a model trained with log loss predicts probability \(\approx 0.8\) means 80% of servers with that prediction are truly over-provisioned.

Quantile Loss (Pinball Loss)

Quantile Loss Formula: $$L_\tau(y, \hat{y}) = \begin{cases} \tau(y - \hat{y}) & \text{if } y \geq \hat{y} \\ (1-\tau)(\hat{y} - y) & \text{if } y < \hat{y} \end{cases}$$

Also written as: \(L_\tau(y, \hat{y}) = \max[\tau(y-\hat{y}), (\tau-1)(y-\hat{y})]\)

The \(\tau\)-quantile minimizer has the property that approximately \(\tau\)-fraction of actual values fall below it. For example, \(\tau=0.95\) produces a forecast such that 95% of historical data lies below it.

CRITICAL FOR CAPACITY PLANNING

Capacity planning cares about tail risk. You need enough capacity to handle 95th or 99th percentile demand, not the average. Quantile loss is the right choice. Use \(\tau=0.95\) to predict the 95th percentile demand, ensuring only 5% of days are under-provisioned (acceptable SLA risk). MSE/MAE predict the mean, which guarantees 50% under-provisioned days—unacceptable.

Huber Loss

Huber Loss Formula: $$L_\delta(y, \hat{y}) = \begin{cases} \frac{1}{2}(y - \hat{y})^2 & \text{if } |y - \hat{y}| \leq \delta \\ \delta|y - \hat{y}| - \frac{1}{2}\delta^2 & \text{if } |y - \hat{y}| > \delta \end{cases}$$

Huber loss is quadratic for small errors and linear for large errors, combining MSE and MAE benefits. It's robust to outliers (doesn't penalize them quadratically) while remaining differentiable everywhere.

1.3 Confusion Matrix & Metrics

Confusion Matrix & Definitions

For a binary classifier predicting class 0 vs class 1:

Predicted Positive (1) Predicted Negative (0)
Actual Positive (1) TP (True Positive) FN (False Negative)
Actual Negative (0) FP (False Positive) TN (True Negative)

Standard Metrics

Precision: $$\text{Precision} = \frac{TP}{TP + FP}$$

Of predictions of class 1, how many are correct? (Positive Predictive Value)

Recall (Sensitivity): $$\text{Recall} = \frac{TP}{TP + FN}$$

Of actual class 1 samples, how many did we find? (True Positive Rate)

F1-Score (Harmonic Mean): $$\text{F1} = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}$$

Balanced measure when classes are imbalanced

Accuracy: $$\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}$$

Overall fraction of correct predictions (misleading with class imbalance)

ROC Curve & AUC

A classifier produces a probability \(p(\mathbf{x})\). Varying a threshold \(\theta \in [0, 1]\), we predict class 1 if \(p(\mathbf{x}) > \theta\). For each threshold, compute:

  • True Positive Rate (TPR): \(\text{TPR} = \frac{TP}{TP+FN}\) (Recall)
  • False Positive Rate (FPR): \(\text{FPR} = \frac{FP}{FP+TN}\) (Specificity)

Plotting TPR vs FPR as \(\theta\) varies gives the ROC curve. AUC (Area Under Curve) measures the probability that the model ranks a random positive example higher than a random negative example. AUC = 0.5 means no discrimination (random classifier); AUC = 1 means perfect separation.

PR Curve & AUC-PR

Precision-Recall curve plots Precision vs Recall as threshold varies. More informative than ROC when classes are imbalanced (positive class is rare). AUC-PR is the area under this curve. A baseline random classifier has AUC-PR equal to the positive class frequency, whereas ROC baseline is always 0.5.

Cloud Context: Cost of Errors

False Negative = OUTAGE RISK

If your classifier predicts a server is NOT over-provisioned (negative) but it actually is (positive class = over-provisioned), you miss an opportunity to reduce costs. Far worse: if your classifier predicts capacity is SUFFICIENT when it's actually INSUFFICIENT (FN in capacity model), users experience outage. This is catastrophic.

False Positive = Overprovision Cost

If your classifier predicts over-provisioned (positive) when the server is actually utilized (negative), you wastefully maintain expensive hardware. This costs money but not customer-facing impact.

Asymmetric Costs: FN (missed under-provisioning) is far more costly than FP (wasted capacity). This breaks the assumption that accuracy or F1-score is the right metric. Instead, choose a threshold that prioritizes high recall at acceptable precision, or use cost-weighted metrics.

Threshold Selection for Business Metrics

Threshold Matters More Than AUC

A model with AUC=0.92 is useless if you pick the wrong threshold. The default threshold of 0.5 is arbitrary and rarely optimal in practice. Example:

  • Threshold 0.5: 80% recall, 60% precision (catch most under-provisions, but flag some false alarms)
  • Threshold 0.8: 50% recall, 95% precision (miss half the under-provisions, but almost never false alarm)

For capacity, choose threshold to achieve ≥95% recall (catch almost all under-provisions) even if precision drops to 30% (overprovision some servers). The business cost of outage outweighs overprovision cost.

1.4 Bias-Variance Tradeoff

Bias-Variance Decomposition

The expected prediction error for a regression model can be decomposed as:

Bias-Variance Decomposition: $$E[(y - \hat{f}(\mathbf{x}))^2] = \underbrace{\text{Bias}^2(\hat{f})}_{\text{underfitting}} + \underbrace{\text{Variance}(\hat{f})}_{\text{overfitting}} + \underbrace{\sigma^2}_{\text{irreducible}}$$

where the expectation is over training sets (i.e., how much \(\hat{f}\) changes with different training data).

  • Bias: How far \(E[\hat{f}(\mathbf{x})]\) is from the true function \(f(\mathbf{x})\). High bias = systematically wrong predictions (underfitting). Example: fitting a line to curved data.
  • Variance: How much \(\hat{f}(\mathbf{x})\) varies across different training sets. High variance = model is sensitive to training noise (overfitting). Example: a degree-20 polynomial that wiggles through every data point.
  • Irreducible Error: Noise in the data itself; cannot reduce this.

Model Complexity vs Bias-Variance

Underfitting (High Bias): Simple models (linear, shallow tree) assume the true relationship is simple. If reality is nonlinear, they systematically mispredict. Training and test error are both high.

Overfitting (High Variance): Complex models (deep trees, many parameters) fit training noise. Training error is low, but test error is high.

Sweet Spot: A model of moderate complexity that balances bias and variance, minimizing test error.

Cloud Capacity Example

A linear trend forecaster (high bias) will consistently underpredict during traffic spikes. A neural network with 1000 hidden units trained on 1 year of daily data (high variance) will overfit to noise in that particular year, failing to generalize to next year. Gradient boosting with moderate tree depth (depth=5, 100 trees) balances both.

Regularization Techniques

L1 & L2 Regularization: Add penalty to loss function:

  • L2: \(\lambda \sum w_i^2\) → shrinks all weights evenly
  • L1: \(\lambda \sum |w_i|\) → zeros out unimportant weights

Dropout: During training, randomly set some activations to zero with probability p. Forces the network to learn redundant representations, reducing co-adaptation.

Early Stopping: Monitor validation loss during training. Stop when validation loss stops improving. Prevents overfitting by not training too long.

Tree Pruning: Remove branches from decision trees that don't improve validation accuracy. Post-training technique.

Cross-Validation: Use multiple train/validation splits to estimate generalization error more reliably.

K-Fold Cross-Validation

Partition data into k folds. For i=1 to k:

  1. Train on folds 1..k-1 (excluding fold i)
  2. Evaluate on fold i
  3. Record test error

Average the k test errors to estimate generalization performance. Typical k=5 or k=10. For time series, use time-aware folds (fold 1 is early dates, fold k is recent dates) to respect temporal dependence.

Visual Intuition

As model complexity increases:

  • Bias decreases (can fit more complex patterns)
  • Variance increases (more sensitive to training noise)
  • Total error (bias² + variance) has a U-shape minimum at optimal complexity
  • Training error decreases monotonically
  • Test error decreases then increases (the valley is where we want to operate)

1.5 Feature Engineering

Lag Features

For time series, the past often predicts the future. Create lagged versions of the target:

Lag Features: $$\mathbf{x}_t = [y_{t-1}, y_{t-2}, \ldots, y_{t-k}]$$

Use autoregressive models: \(\hat{y}_t = f(y_{t-1}, y_{t-2}, \ldots, y_{t-k})\). The choice of k (lag window) depends on the data frequency and dominant cycle. For hourly cloud metrics, k=24 (last 24 hours) captures daily seasonality.

iCloud Capacity Forecasting

Feature set includes: [CPU_t-1, CPU_t-2, ..., CPU_t-24, Memory_t-1, ..., Memory_t-24, RequestRate_t-1, ..., RequestRate_t-24]. The model learns that high CPU usage 2 hours ago predicts high CPU now, and that weekend patterns differ from weekday.

Rolling Aggregates

Summarize recent windows of data:

Rolling Statistics: $$\text{RollingMean}_{t, w} = \frac{1}{w}\sum_{i=0}^{w-1} y_{t-i}$$ $$\text{RollingStd}_{t, w} = \sqrt{\frac{1}{w}\sum_{i=0}^{w-1} (y_{t-i} - \text{RollingMean}_{t,w})^2}$$

Typical windows: last 24 hours (daily), last 7 days (weekly), last 30 days (monthly). Rolling statistics capture volatility and trend:

  • Rolling mean: recent average (trend)
  • Rolling std: recent volatility (risk)
  • Rolling min/max: range of variation

Interaction Features

Combine features to capture nonlinear relationships:

Interaction Example: $$x_{\text{new}} = x_1 \times x_2 \quad \text{or} \quad x_{\text{new}} = \frac{x_1}{x_2}$$

Example: CPU utilization is high, but memory pressure is low. The combination might indicate that only certain workloads are running. Create \(\text{cpu\_to\_memory\_ratio} = \frac{\text{CPU}}{1 + \text{Memory}}\) to capture this.

Note: Gradient boosting trees automatically learn interactions, so manual interaction features may be redundant. However, they help linear models and can improve interpretability.

Encoding Categorical Features

One-Hot Encoding: For a categorical with categories {A, B, C}, create binary columns [is_A, is_B, is_C]. Works well with linear models and trees. Careful with high-cardinality features (100+ categories) as it explodes feature count.

Target Encoding: Replace each category with the average target value for that category. Example: day_of_week → average CPU utilization on Mondays, Tuesdays, etc. Powerful for gradient boosting but risk of leakage if not careful (compute on training set only).

Ordinal Encoding: Assign integers to categories when order matters (e.g., severity levels: low=1, medium=2, high=3).

Feature Leakage Detection

Data Leakage = Invalid Model

Leakage occurs when information from the future or labels leaks into training features. Example: predicting next hour's CPU by including current hour's CPU measured 59 minutes into the hour. The current measurement already reveals the outcome, making the model useless in practice.

Common Leakage Sources:

  • Using future values: \(x_t\) includes data from \(t + 1, t + 2, \ldots\)
  • Label-derived features: feature computed from the target variable itself
  • Data snooping: using test set to select features or hyperparameters
  • Temporal leakage: information that would not be available at prediction time

Prevention: Strictly enforce train/test split timeline. Features used at prediction time must be computable with information available at that time.

Feature Importance & Selection

Permutation Feature Importance: Shuffle each feature's values and measure how much model performance drops. Features with large drops are important. Model-agnostic.

SHAP (SHapley Additive exPlanations) Values: Game-theoretic approach to compute each feature's contribution to each prediction. Provides local and global importance. Better than permutation importance for correlated features.

Feature Selection Methods:

  • Univariate: select top-k features most correlated with target
  • Recursive Elimination: train model, remove weakest feature, repeat
  • L1 Regularization: let Lasso zero out unimportant weights automatically
Feature Engineering Checklist for Cloud Forecasting
  • ☐ Lag features for main metrics (CPU, Memory, Network) at 1h, 6h, 24h, 7d lags
  • ☐ Rolling statistics (mean, std, min, max) over 24h, 7d, 30d windows
  • ☐ Calendar features: hour_of_day, day_of_week, month, is_weekend, is_holiday
  • ☐ External covariates: prior scheduled maintenance, known traffic events
  • ☐ Interaction features: CPU × memory pressure, peak hour × region
  • ☐ Check for leakage: no future information in features
  • ☐ Check for missing values: impute or drop appropriately
  • ☐ Normalize/scale features as needed for the model
  • ☐ Compute feature importance; remove uninformative features
  • ☐ Time series CV: do not shuffle; respect temporal order

Statistical Foundations for Forecasting Time Series Analysis

Time series forecasting requires understanding stationarity, autocorrelation, and the unique challenges of temporal data. These foundations underpin both classical (ARIMA) and modern (ML) forecasting approaches.

2.1 Stationarity

Definitions: Strict vs Weak Stationarity

Strict (Strong) Stationarity: The joint distribution of \((y_t, y_{t+1}, \ldots, y_{t+k})\) is independent of t for all k. In other words, shifting time by any amount doesn't change the statistical properties. Very restrictive in practice.

Weak (Covariance) Stationarity: Three conditions must hold for all t:

  1. Mean is constant: \(E[y_t] = \mu\) (not drifting up or down)
  2. Variance is constant: \(\text{Var}(y_t) = \sigma^2\) (not growing or shrinking)
  3. Autocovariance depends only on lag, not time: \(\text{Cov}(y_t, y_{t+k}) = \gamma_k\) (relationship between past and future is stable)

Why It Matters for Forecasting: Many classical forecasting models (ARIMA, exponential smoothing) assume weak stationarity. If the series is non-stationary,:

  • Model parameters are biased and unreliable
  • Statistical tests (significance, confidence intervals) are invalid
  • Forecasts far into the future become unstable

Visual Examples: Stationary vs Non-Stationary

Stationary Example: CPU Utilization (Daily Mean)

iCloud CPU utilization oscillates around a mean of 65%, staying between 55-75% year-round. No long-term drift. Autocorrelation decays to zero as lag increases. Ready for ARIMA modeling.

Non-Stationary Example: Storage Used

Storage used by iCloud customers grows monotonically from 2020 onward due to increased user base and larger file sizes. Strong upward trend. Variance increases over time (more users = more volatile daily additions). Fails all three stationarity tests. Must difference or detrend first.

Augmented Dickey-Fuller (ADF) Test

The ADF test checks the null hypothesis: "the series has a unit root" (non-stationary). The test regresses:

ADF Regression: $$\Delta y_t = \alpha + \beta t + \gamma y_{t-1} + \sum_{i=1}^{p} \delta_i \Delta y_{t-i} + \epsilon_t$$

The test statistic is \(t_\gamma = \frac{\gamma}{\text{SE}(\gamma)}\). If \(\gamma\) is close to 0 (not significantly negative), we fail to reject the null: the series is non-stationary. If \(\gamma\) is significantly negative, we reject the null: the series is stationary.

Interpretation:

  • p-value < 0.05: Reject null → series is stationary
  • p-value ≥ 0.05: Fail to reject null → series is non-stationary (has unit root)

KPSS Test

KPSS reverses the null hypothesis: "the series is stationary." Complementary to ADF.

  • p-value < 0.05: Reject null → series is non-stationary
  • p-value ≥ 0.05: Fail to reject null → series is stationary

Best Practice: Run both ADF and KPSS. Agreement is strongest evidence. Disagreement (ADF says stationary, KPSS says non-stationary) suggests a slow trend—apply light differencing.

Making Series Stationary: Differencing

First Differencing: Remove level shift by taking differences:

First Difference: $$\Delta y_t = y_t - y_{t-1}$$

Removes linear trends. If \(y_t\) has a constant upward trend, \(\Delta y_t\) will be stationary.

Seasonal Differencing: Remove seasonal patterns by taking differences at the seasonal lag (e.g., lag 24 for hourly data with daily seasonality, lag 7 for daily data with weekly seasonality):

Seasonal Difference (lag m): $$\Delta_m y_t = y_t - y_{t-m}$$

Log Transform: If variance increases with level, apply log first:

Log-Differencing: $$y_t' = \log(y_t), \quad \Delta y_t' = \log(y_t) - \log(y_{t-1}) = \log\left(\frac{y_t}{y_{t-1}}\right) \approx \text{percent change}$$

This captures relative growth rates rather than absolute growth, more stable for heteroscedastic data.

Cloud Example: Storage Forecasting

Storage used (GB) grows from 2020 to 2025, failing stationarity tests. Apply first difference: \(\Delta y_t\) = daily added storage (GB). This is more stable—users add roughly 500GB/day on average with random fluctuations. Now ADF test passes, and ARIMA(p,1,q) models apply.

2.2 Autocorrelation & Seasonality

Autocorrelation Function (ACF)

ACF at Lag k: $$\rho_k = \frac{\text{Cov}(y_t, y_{t-k})}{\text{Var}(y_t)}$$

ACF measures the correlation between \(y_t\) and \(y_{t-k}\). Range: [-1, 1]. The ACF plot shows \(\rho_k\) for k=1, 2, 3, ..., typically with confidence bands (±1.96/√n).

Interpretation:

  • Slow decay: strong dependence on past (autoregressive series)
  • Sharp cutoff at lag q: moving average series MA(q)
  • Spikes at regular intervals (every 24, every 7): seasonality

Partial Autocorrelation Function (PACF)

PACF measures the correlation between \(y_t\) and \(y_{t-k}\) after removing the effects of intermediate lags \(y_{t-1}, y_{t-2}, \ldots, y_{t-k+1}\).

PACF vs ACF Interpretation:

  • ACF decays slowly → AR process (PACF cuts off at lag p)
  • ACF cuts off at lag q → MA process (PACF decays)
  • ACF and PACF both decay → ARMA process (need both AR and MA)

ARIMA Order Selection Using ACF/PACF

ARIMA(p,d,q) structure:

  • p: number of autoregressive terms. If PACF cuts off at lag p, use AR(p).
  • d: number of differences to achieve stationarity. Run ADF test until p < 0.05 (often d=0 or d=1).
  • q: number of moving average terms. If ACF cuts off at lag q, use MA(q).
Practical ARIMA Selection Example

For hourly API request rate:

  1. Plot series → clear daily seasonality and slight upward trend
  2. ADF test on original series → p=0.10 (marginally non-stationary)
  3. Apply first difference → ADF p=0.01 (stationary) → d=1
  4. Plot ACF of differenced series → spikes at lags 1, 24, 48 (seasonality + autoregression)
  5. Plot PACF → significant spike at lag 1 only → p=1
  6. ACF has significant spike at lag 1 → q=1
  7. Use SARIMA(1,1,1)(1,1,1)[24] to capture both hourly AR and daily seasonality

Ljung-Box Test

Test whether autocorrelations at lags 1 to h are jointly significant:

Ljung-Box Statistic: $$Q = n(n+2) \sum_{k=1}^{h} \frac{\rho_k^2}{n-k}$$

Under null (no autocorrelation), Q ~ χ²(h). If p-value < 0.05, at least one autocorrelation is significant. Use to validate that your fitted model has no remaining autocorrelation in residuals (good fit).

Seasonality Detection

In the ACF plot, look for periodic spikes:

  • Hourly data: spike every 24 lags → daily seasonality
  • Daily data: spike every 7 lags → weekly seasonality
  • Hourly API requests: spikes at lags 24 and 168 → daily and weekly seasonality

Cloud Example: iCloud traffic has strong daily seasonality (peak during day, low at night in Pacific timezone) and weak weekly seasonality (slightly lower on weekends). Use SARIMA with seasonal period m=24 (hours) to capture this.

2.3 Bias & Variance in Time Series Forecasting

Forecast Horizon and Error Growth

Forecast error increases with horizon h. For h-step-ahead forecast:

Prediction Variance Growth: $$\text{Var}(\hat{y}_{t+h}) = \sigma^2 \left(1 + \sum_{i=1}^{h-1} \theta_i^2\right)$$

where \(\theta_i\) are MA coefficients. The variance grows with h. At h→∞, an ARIMA model's forecast converges to the mean with unbounded confidence interval.

Recursive vs Direct Multi-Step Forecasting

Recursive (Iterated) Approach: Train a one-step model \(\hat{y}_{t+1} = f(y_t, y_{t-1}, \ldots)\). To forecast h steps ahead:

  1. Predict \(\hat{y}_{t+1} = f(y_t, y_{t-1}, \ldots)\)
  2. Predict \(\hat{y}_{t+2} = f(\hat{y}_{t+1}, y_t, \ldots)\) using predicted values
  3. Continue, feeding predicted values back as inputs

Pros: One model, fast, natural

Cons: Error compounds (bias accumulates); early predictions affect later ones

Direct Approach: Train separate models for each horizon h: \(\hat{y}_{t+h} = f_h(y_t, y_{t-1}, \ldots)\)

Pros: No error accumulation; can use horizon-specific features

Cons: Many models (expensive); no shared learning across horizons

Trade-off: Recursive typically has lower variance but higher bias (especially at long horizons). Direct typically has higher variance but lower bias. For capacity planning where long horizons matter, direct is often preferable despite higher variance.

Conditional Heteroscedasticity

Forecast error variance may change over time (heteroscedasticity). During high-volatility periods (peak traffic), forecasts are less certain. GARCH (Generalized Autoregressive Conditional Heteroscedasticity) models capture this:

GARCH(1,1): $$\sigma_t^2 = \omega + \alpha y_{t-1}^2 + \beta \sigma_{t-1}^2$$

The conditional variance \(\sigma_t^2\) depends on past squared shocks \(y_{t-1}^2\) and past variance \(\sigma_{t-1}^2\). This produces wider prediction intervals during volatile periods—appropriate for risk-aware capacity planning.

2.4 Train/Test Splits for Time Series

Why Random Shuffling Is Fatal

NEVER Random Shuffle Time Series

Time series have temporal dependence. If you shuffle dates, training data from June can "learn" from test data from May, leaking information. The model will appear better than it actually is in production.

Example of Leakage: Shuffle hourly data. Training set contains hour 12:00 from 3 different days; test set contains hour 13:00 from those same days. The model learns that 13:00 follows 12:00 even though those hours were never adjacent in the original time series!

Simple Train/Test Split

Respect chronological order:

  1. Choose cutoff date D
  2. Train: all data before D
  3. Test: all data from D onward

Example: Train on 2023-01-01 to 2024-12-31 (24 months). Test on 2025-01-01 to 2025-02-12 (6 weeks of recent data).

Limitation: Single split; if test set happens to be anomalous (unusual traffic patterns in January), performance estimate is unreliable.

Rolling-Origin (Walk-Forward) Validation

More robust: simulate multiple real-world scenarios by rolling the train/test boundary forward.

Walk-Forward Backtesting Procedure

For each month in the test period:

  1. Train on all data up to month start (expanding window)
  2. Forecast next month
  3. Record errors
  4. Move month boundary forward

This simulates the real deployment scenario: refit the model monthly with the latest data, forecast the next month, evaluate accuracy.

Sliding Window vs Expanding Window

Expanding Window: Keep all historical data. Train on [2020-01-01, 2024-12-31], then [2020-01-01, 2025-01-31], etc. More data → stable estimates but slower retraining.

Sliding Window: Keep fixed-size window. Train on [2024-01-01, 2024-12-31], then [2024-02-01, 2025-01-31], etc. Faster retraining, adapts to recent patterns, but less historical data → higher variance.

For iCloud: Expanding window is safer for capacity planning (we want to learn all regimes: quiet periods, peak seasons, outages). Update model monthly with latest data.

Gap Between Train and Test

If your model uses features up to time t, ensure test period starts from t+1 onward. For example:

  • Feature engineering uses data through 2024-12-31 23:00
  • Test period: 2025-01-01 00:00 onward

A 1-2 day gap is common to ensure no accidental leakage from data replication delays.

2.5 Error Metrics for Time Series Forecasting

MAPE (Mean Absolute Percentage Error)

MAPE: $$\text{MAPE} = \frac{100}{n}\sum_{i=1}^{n} \left|\frac{y_i - \hat{y}_i}{y_i}\right|$$

Expresses error as percentage of actual value. Interpretable: 10% MAPE means average forecast is 10% off.

Problem: Undefined when \(y_i = 0\). Biased: overestimates error when actual is small, underestimates when actual is large. If actual=100, forecast=110 (10% error), but if actual=10, forecast=11 (10% error)—MAPE treats latter worse.

sMAPE (Symmetric MAPE)

sMAPE: $$\text{sMAPE} = \frac{100}{n}\sum_{i=1}^{n} \frac{2|y_i - \hat{y}_i|}{|y_i| + |\hat{y}_i|}$$

Symmetric: treats overestimate and underestimate equally. Still has issues (not scale-invariant, bounded by 200%). Less popular than MAPE despite being "better" statistically.

RMSE (Root Mean Squared Error)

RMSE: $$\text{RMSE} = \sqrt{\frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2}$$

Penalizes large errors quadratically. Units same as target variable (e.g., GB). A single huge forecast error dominates RMSE.

Cloud Use: Penalizes missing capacity spikes. If forecast is 10 GB but actual is 100 GB (storage surge), RMSE = 90 is very high. Good for capturing tail events.

WAPE (Weighted Absolute Percentage Error)

WAPE: $$\text{WAPE} = \frac{\sum_{i=1}^{n}|y_i - \hat{y}_i|}{\sum_{i=1}^{n}|y_i|} \times 100\%$$

Total error divided by total actual. Avoids the zero-division problem of MAPE and is scale-invariant. Example: if actual total is 1000 GB and errors sum to 50 GB, WAPE = 5%.

Best for: Aggregated forecasts (e.g., total data center utilization). Less sensitive to individual outliers.

MAE vs RMSE: Which to Optimize?

MAE (L1 loss):

  • Robust to outliers
  • Less sensitive to tail events
  • Optimizes median forecast

RMSE (L2 loss):

  • Sensitive to large errors
  • Captures outliers/spikes
  • Optimizes mean forecast

Decision for Capacity Planning: RMSE is preferred because underestimating capacity (missing a spike) is worse than overestimating. RMSE's sensitivity to large errors aligns with business priorities.

Quantile Loss for Tail Risk

Quantile Loss (τ=0.95): $$L_{0.95} = \frac{1}{n}\sum_{i=1}^{n} \begin{cases} 0.95(y_i - \hat{y}_i) & \text{if } y_i \geq \hat{y}_i \\ 0.05(\hat{y}_i - y_i) & \text{if } y_i < \hat{y}_i \end{cases}$$

Quantile loss assigns asymmetric penalties: 95% penalty for underestimating (missing capacity), 5% penalty for overestimating (wasted capacity). Naturally leads to forecasts where ~95% of outcomes are below the forecast—exactly what capacity needs.

Choosing Metrics for iCloud

Metric Best For Pros Cons
MAPE Typical accuracy reporting Interpretable (%) Fails at zero; biased
WAPE Aggregated forecasts Interpretable; robust Less sensitive to spikes
RMSE Mean forecast (general purpose) Penalizes spikes; standard Sensitive to outliers
Quantile Loss (0.95) CAPACITY PLANNING Directly optimizes tail Less familiar to teams
Metric Selection Determines Outcome

If you optimize MAPE, the model produces mean forecasts. If you optimize quantile loss with τ=0.95, it produces 95th-percentile forecasts. For capacity planning, the latter prevents outages; the former causes them on 50% of days. Choose your metric based on business requirements, not statistical convention.

Forecasting Methods Classical to Modern ML

From Box-Jenkins ARIMA to transformer-based deep learning, this section covers the forecasting toolbox. Each method has strengths and weaknesses; the key is matching the right tool to the problem and data characteristics.

3.1 Classical Forecasting Methods

ARIMA(p,d,q): Autoregressive Integrated Moving Average

ARIMA combines three components:

  • AR(p): Autoregressive—past values predict future. \(y_t = c + \sum_{i=1}^{p} \phi_i y_{t-i} + \epsilon_t\)
  • I(d): Integrated—difference d times to achieve stationarity. \(\Delta^d y_t\) is stationary
  • MA(q): Moving Average—past errors predict future. \(y_t = c + \epsilon_t + \sum_{j=1}^{q} \theta_j \epsilon_{t-j}\)

Model Selection (Box-Jenkins Methodology):

  1. Plot series; determine d via ADF test
  2. Plot ACF/PACF of differenced series
  3. Identify p and q from ACF/PACF cutoffs
  4. Fit ARIMA(p,d,q)
  5. Check residuals (should be white noise via Ljung-Box)
  6. If residuals autocorrelated, increase p or q; repeat

SARIMA(p,d,q)(P,D,Q)[m]: Seasonal ARIMA

Extends ARIMA with seasonal components. Seasonal period m (e.g., m=24 for hourly data with daily seasonality):

SARIMA Structure: $$\Phi(B^m)\phi(B)\Delta^D \Delta^d y_t = \Theta(B^m)\theta(B)\epsilon_t$$

where \(\phi(B), \theta(B)\) are standard AR/MA polynomials, and \(\Phi(B^m), \Theta(B^m)\) operate at seasonal lags.

Cloud Example: SARIMA(1,1,1)(1,1,1)[24] models hourly API request volume:

  • p=1: current hour depends on last hour
  • d=1: first difference removes hourly trend
  • q=1: current error influenced by last hour's error
  • P=1: current hour depends on same hour yesterday
  • D=1: seasonal difference (24-hour cycle)
  • Q=1: seasonal error influence

Model Selection: AIC/BIC

AIC (Akaike Information Criterion): $$\text{AIC} = 2k - 2\ln(\hat{L})$$

where k is the number of parameters and \(\hat{L}\) is the maximum likelihood. Lower AIC is better. Penalizes model complexity but favors fit.

BIC (Bayesian Information Criterion): $$\text{BIC} = k\ln(n) - 2\ln(\hat{L})$$

Heavier penalty on model complexity (grows with n). BIC prefers simpler models than AIC. Use grid search over (p,d,q) combinations, select model with lowest AIC/BIC.

Exponential Smoothing (ETS)

Simple Exponential Smoothing: Forecast is a weighted average of all past observations, with exponentially decaying weights:

SES Update: $$\ell_t = \alpha y_t + (1-\alpha)\ell_{t-1}$$ $$\hat{y}_{t+h} = \ell_t$$

where \(\ell_t\) is the level and \(\alpha \in (0,1)\) is the smoothing parameter. High \(\alpha\) = recent observations weighted heavily; low \(\alpha\) = smooth, stable forecasts.

Holt's Linear Trend Method: Extends SES to capture trend:

Holt Updates: $$\ell_t = \alpha y_t + (1-\alpha)(\ell_{t-1} + b_{t-1})$$ $$b_t = \beta(\ell_t - \ell_{t-1}) + (1-\beta)b_{t-1}$$ $$\hat{y}_{t+h} = \ell_t + h \cdot b_t$$

where \(\ell_t\) is level, \(b_t\) is trend, and \(\beta\) is the trend smoothing parameter.

Holt-Winters Seasonal: Extends Holt to include seasonal component:

Holt-Winters (Additive Seasonality): $$\ell_t = \alpha(y_t - s_{t-m}) + (1-\alpha)(\ell_{t-1} + b_{t-1})$$ $$b_t = \beta(\ell_t - \ell_{t-1}) + (1-\beta)b_{t-1}$$ $$s_t = \gamma(y_t - \ell_t) + (1-\gamma)s_{t-m}$$ $$\hat{y}_{t+h} = \ell_t + h \cdot b_t + s_{t+h-m}$$

where \(s_t\) is seasonal component and m is seasonal period.

Multiplicative Seasonality: Use when seasonal variation scales with level. Replace additions with multiplications: \(y_t = (\ell_{t-1} + b_{t-1}) \times s_{t-m}\). Better for data where variance grows with level.

Prophet (Facebook/Meta)

Decomposable model: \(y_t = g(t) + s(t) + h(t) + \epsilon_t\)

  • g(t): trend (piecewise linear or logistic growth)
  • s(t): seasonality (Fourier series)
  • h(t): holiday/event effects
  • \(\epsilon_t\): error term

Advantages: Handles missing data, outliers; easy to specify holidays; good for business time series with clear seasonality. Interpretable components.

Disadvantages: Assumes additive decomposition; struggles with complex nonlinear dynamics; slower than ARIMA/ETS.

Cloud Use: Prophet works well for high-level capacity forecasts (weekly/monthly aggregates). For hourly granular forecasting, classical ARIMA or ML methods are better.

When to Use Classical Methods

Classical Methods Excel When:
  • Short forecast horizon (days to weeks)
  • Univariate (one time series to forecast)
  • Stable, linear dynamics with clear seasonality
  • Small data (< 10k observations)
  • Need interpretability (explain forecast to business)
  • Quick baseline needed (ARIMA fits fast)

Limitations of Classical Methods

Classical Methods Struggle With:
  • Multiple input variables (exogenous covariates)
  • Nonlinear relationships (traffic × time_of_day interaction)
  • Regime changes (sudden shift in user behavior)
  • Long forecast horizons (error/interval widens dramatically)
  • Large datasets (computationally expensive)
  • Irregular/sparse data (missing values problematic)

3.2 ML-Based Forecasting Methods

Gradient Boosted Trees with Feature Engineering

Train XGBoost/LightGBM/CatBoost on engineered features from the time series. The model learns nonlinear patterns without assuming a specific functional form.

Feature Set for Cloud Forecasting:

  • Lags: y_{t-1}, y_{t-2}, ..., y_{t-24} (hourly), ..., y_{t-168} (weekly)
  • Rolling Statistics: rolling_mean_{24h}, rolling_std_{24h}, rolling_mean_{7d}, rolling_std_{7d}
  • Calendar: hour_of_day, day_of_week, month, is_weekend, is_holiday
  • External: prior_demand (forecast from other regions), scheduled_events
  • Interactions: hour × is_weekend (traffic patterns differ), peak_hour × region

Advantages:

  • Handles many features automatically (feature importance)
  • Robust to outliers (tree-based)
  • Fast training and prediction
  • Learns nonlinear interactions without manual specification
  • Dominates on tabular cloud metrics data in practice

Disadvantages:

  • Black-box; harder to explain why forecast is X
  • Requires feature engineering (not end-to-end learning)
  • Fixed context window (lags up to t-168); misses very long-range dependencies

In Production: Gradient boosting is the current standard for iCloud capacity forecasting. Fast retraining (minutes), accurate (RMSE 5-10% on hourly data), interpretable feature importance, minimal infrastructure.

LSTM (Long Short-Term Memory) Neural Networks

RNNs process sequences by maintaining hidden state \(h_t\) that captures information from all past inputs. Vanilla RNNs suffer from vanishing gradients (backprop through time loses signal). LSTMs solve this with gating mechanisms:

LSTM Cell (Simplified): $$f_t = \sigma(W_f \cdot [h_{t-1}, x_t] + b_f) \quad \text{(forget gate)}$$ $$i_t = \sigma(W_i \cdot [h_{t-1}, x_t] + b_i) \quad \text{(input gate)}$$ $$\tilde{C}_t = \tanh(W_C \cdot [h_{t-1}, x_t] + b_C) \quad \text{(candidate memory)}$$ $$C_t = f_t * C_{t-1} + i_t * \tilde{C}_t \quad \text{(cell state)}$$ $$o_t = \sigma(W_o \cdot [h_{t-1}, x_t] + b_o) \quad \text{(output gate)}$$ $$h_t = o_t * \tanh(C_t) \quad \text{(hidden state)}$$

Forget Gate: Controls how much past information to forget. High forget gate = rely on old memory.

Input Gate: Controls how much new information to add. High input gate = incorporate new observation.

Output Gate: Controls what hidden state to expose for next step and output.

When LSTM Excels:

  • Very long sequences (1000+ time steps) where distant past matters
  • Complex temporal dependencies (nonlinear, long-range)
  • End-to-end learning (raw time series → forecast without hand-crafted features)
  • Large datasets (>100k samples) to support parameter count

Limitations for Cloud Capacity:

  • Slower training/inference than tree-based models
  • Requires careful tuning (learning rate, regularization)
  • Need GPU for fast training
  • Black-box; no feature importance for interpretability
  • Empirically, not always better than well-tuned gradient boosting for tabular cloud metrics

Temporal Fusion Transformer (TFT)

Attention-based architecture for multi-horizon time series forecasting. Distinguishes between:

  • Static covariates: unchanging (e.g., data center region)
  • Known future inputs: available at forecast time (e.g., scheduled maintenance, calendar)
  • Unknown future inputs: only available in past (e.g., actual CPU utilization)

Key Components:

  • Variable Selection Network: Attention to select relevant features for each time series
  • Temporal Self-Attention: Learn temporal dependencies
  • Quantile Regression Head: Output prediction intervals (0.1, 0.5, 0.9 quantiles)

Advantages:

  • State-of-the-art accuracy on benchmarks
  • Produces prediction intervals natively (good for capacity tail forecasts)
  • Interpretable attention weights (which time steps matter?)
  • Handles heterogeneous multivariate input (mix of raw and engineered features)

Disadvantages:

  • Complex architecture; many hyperparameters to tune
  • Requires large dataset (100k+ samples) and GPU training
  • Longer inference latency than trees
  • Library maturity (PyTorch Forecasting) not as stable as sklearn/XGBoost

For iCloud: TFT is a strong candidate for next-generation capacity forecasting if you have resources for model ops. Currently reserved for research/advanced use cases; gradient boosting remains production standard.

3.3 Multi-Horizon Forecasting Strategies

Capacity planning needs forecasts at multiple horizons: minutes (autoscaling), hours (load balancing), days (resource allocation), months (procurement). How to efficiently forecast many steps ahead?

Direct Approach: Separate Model per Horizon

Train model \(f_h\) for each horizon h ∈ {1, 6, 24, 168} (1 hour, 6 hours, 1 day, 1 week):

Direct Forecasting: $$\hat{y}_{t+h} = f_h(y_t, y_{t-1}, \ldots, y_{t-k}), \quad \text{for each } h$$

Advantages:

  • No error accumulation: each model predicts directly from observed inputs
  • Horizon-specific features (e.g., features engineered just for 24h horizon)
  • Parallelizable (train all models in parallel)

Disadvantages:

  • Many models (maintenance burden)
  • No sharing of learning across horizons
  • Inconsistent: \(\hat{y}_{t+1} + \hat{y}_{t+2} \neq \hat{y}_{t+2}|_{h=2}\) (one-step sum differs from direct two-step)

Recursive (Iterated) Approach: One Model for All Horizons

Train one-step model \(f_1\), use it recursively:

Recursive Forecasting: $$\hat{y}_{t+1} = f_1(y_t, y_{t-1}, \ldots, y_{t-k})$$ $$\hat{y}_{t+2} = f_1(\hat{y}_{t+1}, y_t, \ldots, y_{t-k+1})$$ $$\hat{y}_{t+h} = f_1(\hat{y}_{t+h-1}, \ldots, \hat{y}_{t+h-k})$$

Advantages:

  • One model; simple, fast to train
  • Consistent: multi-step forecast from sum of one-step forecasts

Disadvantages:

  • Error compounds: \(E[\hat{y}_{t+h}]\) can have h times the bias of one-step error
  • Uses own predictions as inputs → distribution mismatch (training on true values, predicting with model predictions)
  • Longer horizons become increasingly unreliable

For Cloud Capacity: Direct approach is safer for important horizons (procurement). Recursive can handle shorter horizons (real-time autoscaling).

Multi-Output Approach: Single Model, All Horizons Simultaneously

Train a model with multiple output heads, each predicting one horizon:

Multi-Output Model: $$[\hat{y}_{t+1}, \hat{y}_{t+6}, \hat{y}_{t+24}, \hat{y}_{t+168}] = f(\text{features}_t)$$

Architecture Examples:

  • Seq2Seq (Encoder-Decoder): Encode input sequence, decode to multiple outputs. Used by deep learning methods.
  • TFT: Single attention mechanism, multiple quantile heads (0.1, 0.5, 0.9)
  • Gradient Boosting with Multi-Output: Train single tree ensemble, apply to multiple targets. Less common but feasible.

Advantages:

  • Shared learning across horizons (better generalization)
  • No error accumulation (direct)
  • One model per task

Disadvantages:

  • More complex training (multiple losses to balance)
  • Predictions must be reconciled (e.g., ensure 24h forecast = sum of 1h forecasts?)

iCloud Multi-Horizon Strategy

Recommended Approach

Use direct models for strategic horizons:

  • 5-minute to 1-hour horizon: Recursive tree-based model (fast, stable)
  • 1-day horizon: Direct gradient boosting model (procurement planning)
  • 1-week horizon: Direct model (hardware allocation)
  • 1-month+ horizon: Direct model with trend component (procurement lead time)

This balances simplicity (few models) with reliability (direct for important decisions).

Variance Growth with Horizon

Regardless of method, forecast uncertainty grows:

Variance vs Horizon: $$\text{Var}(\hat{y}_{t+h}) \approx \sigma^2 (1 + c \cdot h)$$

where c depends on model/data. At very long horizons (months ahead), confidence intervals become so wide they're useless. This is fundamental: no amount of data or modeling fixes it. Plan accordingly in capacity forecasts.

3.4 Forecasting Tails: Critical for Capacity Planning

Why Means Are Not Enough

Mean Forecast = 50% Underprovisioned Days

A forecast of the average (mean) demand guarantees that half the days actual demand exceeds forecast. If your forecast is the mean, you're under-provisioned 50% of the time. For iCloud, this is unacceptable (SLA violation).

Business Requirement: Provision for 95th-percentile or 99th-percentile demand. This ensures 95% or 99% of days have sufficient capacity, accepting only 5% or 1% underprovisioning risk.

Quantile Regression: Regression for Any Percentile

Instead of predicting E[y|x] (mean), predict Q_τ(y|x) (τ-quantile):

Quantile Loss Minimization: $$\arg\min_{\hat{q}_\tau} \frac{1}{n}\sum_{i=1}^{n} L_\tau(y_i, \hat{q}_\tau(\mathbf{x}_i))$$

where

Pinball Loss (Quantile Loss): $$L_\tau(y, \hat{q}) = \begin{cases} \tau(y - \hat{q}) & \text{if } y \geq \hat{q} \\ (1-\tau)(\hat{q} - y) & \text{if } y < \hat{q} \end{cases}$$

Interpretation: The loss function asymmetrically penalizes errors. Underestimating (actual > forecast) costs \(\tau\) per unit; overestimating costs \((1-\tau)\) per unit. Minimizing this loss produces forecasts where \(\tau\)-fraction of data lies below the forecast.

Example: τ=0.95 gives equal loss if we underestimate by 100 or overestimate by 20 (since 0.95×100 = 0.05×2000... actually this doesn't balance; let me recalculate). If τ=0.95, underestimating by 1 costs 0.95; overestimating by 1 costs 0.05. We weight underestimation 19x more, so the optimizer prefers to err on the high side—resulting in forecasts where ~95% of actuals fall below.

Quantile Regression with Gradient Boosting

Most boosting libraries (XGBoost, LightGBM, CatBoost) natively support quantile loss:

# XGBoost Quantile Regression import xgboost as xgb model = xgb.XGBRegressor(objective='reg:quantilehubererror', quantile_alpha=0.95) model.fit(X_train, y_train) forecast_p95 = model.predict(X_test) # 95th percentile forecast

Train multiple models for multiple quantiles: τ ∈ {0.05, 0.50, 0.95} gives prediction interval [5th, 95th percentile].

Conformal Prediction: Distribution-Free Prediction Intervals

Conformal prediction wraps any point estimator (tree, neural network, etc.) to produce prediction intervals with finite-sample coverage guarantees. Key property: makes minimal assumptions about the data distribution.

Procedure (Regression):

  1. Split data into fit set (train model) and calibration set
  2. Train predictor \(\hat{f}\) on fit set
  3. On calibration set, compute residuals: \(e_i = |y_i - \hat{f}(x_i)|\)
  4. For new test point, compute prediction interval as: \(\hat{f}(x) \pm \hat{q}_{1-\alpha}\)
  5. where \(\hat{q}_{1-\alpha}\) is the \((1-\alpha)\)-quantile of calibration residuals

Guarantee: At least \((1-\alpha)\)-fraction of new test points will have actual values in the predicted interval. This holds for ANY predictor and ANY data distribution (without assuming normality, independence, etc.).

Cloud Application: Wrap your gradient boosting model with conformal prediction. Get 95% coverage intervals guaranteed, without distributional assumptions. Perfect for capacity planning where you need hard guarantees.

Extreme Value Theory (EVT): Modeling Tail Behavior

For rare, extreme events (traffic spikes, data center failures), the tail of the distribution matters. EVT models extreme values using the Generalized Pareto Distribution (GPD).

Peaks-Over-Threshold Approach:

  1. Set threshold u (e.g., 90th percentile of historical data)
  2. Extract all exceedances: X = {y : y > u}
  3. Fit GPD to exceedances: \(F(x|x>u) = 1 - (1 + \xi x/\sigma)^{-1/\xi}\)
  4. Estimate extreme quantiles (99.9th percentile) using fitted GPD

Use Case: Historical data shows traffic spikes up to 10x mean during major events. To estimate 99.9th percentile capacity need, fit EVT model to exceedances above the 90th percentile and extrapolate.

Limitations: Requires careful threshold selection. Too low → not capturing tail; too high → too few samples. Often used to augment quantile regression methods.

Practical Approach: Quantile Regression + Conformal Calibration

Recommended Stack for iCloud Capacity
  1. Train gradient boosting model with quantile loss (τ=0.95)
  2. Wrap with conformal prediction for finite-sample coverage guarantee
  3. Monitor calibration (actual 95% coverage?) weekly; adjust if needed
  4. Use to set capacity targets: provision for conformal 95% interval upper bound

3.5 Calibration & Coverage

Calibration: Definition & Importance

A p95 forecast is calibrated if 95% of actual observations fall below it.

Calibration (Empirical Coverage): $$\text{Coverage}_\tau = \frac{1}{n}\sum_{i=1}^{n} \mathbb{1}(y_i \leq \hat{q}_{\tau,i}) \approx \tau$$

where \(\mathbb{1}\) is the indicator function.

Example: You produce 95th percentile forecasts for CPU utilization over 100 days. On 94 days, actual CPU ≤ forecast; on 6 days, actual CPU > forecast. Empirical coverage = 94/100 = 0.94 ≈ 0.95 ✓ Calibrated.

Under-Coverage vs Over-Coverage

Under-Coverage: Coverage < target (actual outliers exceed forecast). Example: produce p95 forecasts but only 90% of actuals fall below.

Under-Coverage = Outage Risk

If your p95 forecast undercovers, more than 5% of days have capacity shortage. SLA violations, outages, customer impact.

Over-Coverage: Coverage > target (forecast too high). Example: produce p95 forecasts but 99% of actuals fall below.

Over-Coverage = Wasted Capacity

Overly conservative forecasts waste expensive hardware. Fine for safety, but costs money. Goal: calibrate exactly right.

Sharpness: Among Calibrated Forecasts, Prefer Tight Intervals

Two models produce well-calibrated 95% intervals:

  • Model A: predicts [low, high] with (high - low) = 10 GB
  • Model B: predicts [low, high] with (high - low) = 15 GB

Both have 95% coverage, but Model A is sharper (tighter interval). Prefer A: it gives more actionable guidance (provision 10 GB more) vs Model B (provision 15 GB more).

Definition: Sharpness = average interval width. Lower is better (more precise guidance).

Calibration Plot: Diagnostic Tool

Plot predicted quantile on x-axis vs observed frequency on y-axis. For τ = 0.1, 0.25, 0.50, 0.75, 0.90, 0.95:

  • x = 0.1 (10th percentile forecast), y = fraction of data below it
  • If perfectly calibrated: y = 0.1
  • If y > 0.1: over-coverage (too conservative)
  • If y < 0.1: under-coverage (too optimistic)

Plot points should lie on the diagonal line (y=x). Deviation indicates miscalibration.

PIT (Probability Integral Transform) Histogram

For distributional forecasts (e.g., quantile regression with multiple quantiles), check if the predicted CDF is well-calibrated.

Procedure:

  1. For each test sample, compute the predicted CDF at the actual value: \(\hat{F}_i(y_i)\)
  2. Collect all PIT values: \(\{\hat{F}_1(y_1), \hat{F}_2(y_2), \ldots, \hat{F}_n(y_n)\}\)
  3. Plot histogram of PIT values

Interpretation: If forecasts are well-calibrated, PIT values should be uniformly distributed on [0,1]. Histogram should be flat. Spikes indicate miscalibration (actual data clusters in certain probability regions).

How to Fix Miscalibration: Isotonic Regression Recalibration

If forecasts are miscalibrated (e.g., 95% forecasts only have 92% coverage), recalibrate post-hoc:

  1. On validation set, compute (predicted quantile, observed indicator) pairs
  2. Fit isotonic regression: \(\text{obs} = f_{\text{iso}}(\text{pred})\) (monotonic function)
  3. On test set, apply transformation: \(\hat{q}_{\text{new}} = f_{\text{iso}}^{-1}(\text{pred})\)

Isotonic regression finds the best monotonic function fitting the validation data. After recalibration, coverage should match target.

Calibration Checklist for Capacity Forecasting
  • ☐ Train quantile regression model (τ=0.95 for capacity)
  • ☐ On validation set, compute empirical coverage
  • ☐ If coverage ≠ 0.95, apply isotonic recalibration
  • ☐ On test set, verify coverage again (should match 0.95)
  • ☐ Plot calibration curve (predicted vs observed quantiles)
  • ☐ Monitor coverage in production (daily/weekly)
  • ☐ If coverage drifts (seasonality changes, regime shift), retrain and recalibrate
  • ☐ Document interval width (sharpness) for business reporting
  • ☐ Integrate with procurement/autoscaling systems to use forecasts

Conformal Calibration: Guaranteed Coverage

Even after recalibration, coverage is empirical (observed on validation set). For guaranteed coverage, use conformal prediction:

Conformal Setup: Any point predictor + conformal wrapper → prediction interval guaranteed \((1-\alpha)\) coverage on new test data (without distributional assumptions).

Use Conformal Prediction for Hard Guarantees

If you need mathematical certainty that your 95% capacity forecast covers 95% of days (not just empirically, but theoretically), use conformal prediction. The guarantee holds for ANY data distribution and ANY model.

The fundamental constraint in hierarchical forecasting: aggregation property. If you forecast demand at multiple levels, forecasts at all child nodes must sum to the parent forecast.

Without enforcing coherency, you get inconsistent decisions: autoscaler scales up a cluster, but regional budget forecasts under-provision network egress.

4.2 Reconciliation Methods

Since you can forecast at any level (top, bottom, middle), and each approach has trade-offs, the question becomes: how do you combine forecasts to maintain coherency?

Bottom-Up Approach

Forecast at the leaf nodes (finest granularity), then sum upward.

Pros Cons
Captures local patterns, seasonality unique to each cluster High noise in leaf forecasts propagates; variability compounds
No information loss; each node gets specialized model Computationally expensive (forecast 1000s of hosts individually)
Naturally incorporates local dynamics (sudden traffic shifts) May miss macro trends or correlations across regions

When to use: When leaf-level data is rich and stable (e.g., large clusters with consistent traffic). Typical for 7-14 day forecasts where local dynamics matter.

Top-Down Approach

Forecast at the top level, then disaggregate using proportions or drivers.

Pros Cons
Smooth, stable aggregate forecasts; macro trends dominate Ignores local dynamics, sudden shifts in cluster demand
Simple to compute; one model at the top Proportions may not reflect current data; stale disaggregation
Inherently coherent by construction Loses valuable information about where demand actually is

Disaggregation methods:

1. **Proportional:** Allocate top forecast using historical proportions child_forecast = top_forecast × (historical_child_prop) 2. **Driver-based:** Use known drivers (user growth by region, product mix) child_forecast = top_forecast × (current_market_share) 3. **Average historical:** Split equally across all children

When to use: When bottom levels are sparse or noisy. Common for long-term (30-90 day) strategic forecasts. Also useful when you have strong top-level trend but weak leaf-level signals.

Middle-Out Approach

Forecast at a strategically chosen middle level (e.g., per-cluster instead of per-host), then reconcile both up and down.

Cluster level strikes a good balance: enough data to be stable, still captures geographic and service-type variation. Reconcile up (sum clusters to regions), then allocate down (cluster forecast to individual hosts) if needed.

When to use: When you have clear natural "planning units" (e.g., clusters). Combines benefits of bottom-up (local signal) and top-down (stability).

MinT (Minimum Trace) Reconciliation

Instead of choosing just one level, forecast at all levels independently, then find an optimal linear combination that preserves coherency.

MinT Formula: $$\tilde{\mathbf{y}} = \mathbf{S}(\mathbf{S}'\mathbf{W}^{-1}\mathbf{S})^{-1}\mathbf{S}'\mathbf{W}^{-1}\hat{\mathbf{y}}$$

Where:

  • \(\mathbf{S}\) = summing matrix (encodes the hierarchy structure)
  • \(\hat{\mathbf{y}}\) = base forecasts at all levels
  • \(\mathbf{W}\) = error covariance matrix from base forecasts
  • \(\tilde{\mathbf{y}}\) = reconciled, coherent forecasts

Intuition: Weight each base forecast by the inverse of its error variance. Forecasts with low variance (high quality) get higher weight. The result is guaranteed coherent and optimal (minimum variance).

OLS Reconciliation (Simplified MinT)

When estimating \(\mathbf{W}\) is hard, use OLS: set \(\mathbf{W} = \mathbf{I}\) (identity matrix). Simpler, less data-hungry, often works well in practice.

Practical Challenges with MinT

Covariance estimation: Estimating \(\mathbf{W}\) requires sufficient history and rank-deficiency handling. For 1000+ nodes, the matrix is massive.

Shrinkage: Use shrinkage estimators (Ledoit-Wolf) to stabilize the covariance estimate.

Computation: Matrix inversion is O(n³). For large hierarchies, use approximations or block-wise reconciliation.

Recommendation Framework

When to Use Each Approach

Bottom-Up: Short horizon (1-7 days), rich leaf data, stable operational forecasts.

Top-Down: Long horizon (30+ days), sparse bottom levels, strategic capacity planning.

Middle-Out: Medium horizon (7-30 days), natural planning units, tactical decisions.

MinT/OLS: When you have forecasts at multiple levels and want optimal coherent combination. Requires enough historical data. Good for ensemble approaches.

4.3 Spatial Correlation and Geographic Effects

Demand doesn't move in isolation. When a new product launches (e.g., new iPhone), photo upload demand spikes first in US time zones, then propagates to Europe, then Asia. Geographic proximity and network effects create spatial structure in demand.

Sources of Spatial Correlation

Mechanism Example Forecast Impact
Event propagation iPhone launch photo uploads spread across time zones 6-12 hour lag between regions; forecast one region to predict the next
Traffic routing/spillover US-east data center overloaded → traffic routes to us-west Demand in one region influences neighbor; need coupled models
CDN/edge caching Popular content cached in regional edge nodes Forecast where content will be consumed, not origin demand
User behavior timezone Morning backup surge in Asia, evening in Europe Time of day is proxy for geography; capture with timezone features

Modeling Spatial Correlation

Approach 1: Geographic Covariates

Include features encoding location in your forecasting model:

# Feature engineering for spatial modeling features = [ region_id (categorical), latitude, longitude, distance_to_major_cities, timezone, population_density, user_base_in_region, peak_hours_local_timezone, ]

Simple and effective: if your model is a gradient boosting tree or neural net, it can learn that Europe follows US by 6-8 hours.

Approach 2: Spatial Autoregressive Models (SAR)

SAR Model: $$y_t = \rho \mathbf{W} \mathbf{y}_t + \mathbf{X}_t \boldsymbol{\beta} + \boldsymbol{\epsilon}_t$$

Where \(\mathbf{W}\) is a spatial weight matrix (e.g., inverse distance between regions), and \(\rho\) is the spatial autocorrelation coefficient. Higher \(\rho\) means nearby regions' demand strongly influences your region.

Pros: Explicitly models spillover. Cons: More complex estimation, harder to interpret, less scalable.

Approach 3: Graph Neural Networks (GNN)

Represent data centers as nodes in a graph, network links as edges. Use graph convolution to aggregate spatial information:

# Pseudo-code: spatial aggregation for each region r: neighbor_demand = aggregate(demand[neighboring_regions]) region_forecast[r] = model( own_features[r], own_history[r], neighbor_demand )

Modern approach, captures complex spatial patterns. Requires sufficient training data and careful design.

Practical Spatial Features for iCloud

Geographic Features Engineering

Timezone and time-of-day interaction: Use local hour (e.g., 0-23 in region's timezone), not UTC. Peak backup times are 2am-4am local time globally.

Regional user growth: Include lagged user additions by region (from analytics). Drives sustained demand growth.

Distance to major hubs: Compute haversine distance to NYC, London, Tokyo. Relevant for latency-sensitive services.

Seasonal/holiday calendars: Different holidays in different regions. US Thanksgiving ≠ UK Boxing Day.

Product rollout timing: New features often launch in one region, then roll out. Include "days since launch in region" as a feature.

4.4 Spatio-Temporal Forecasting Models

The previous section treated space and time separately. Spatio-temporal models explicitly optimize for both dimensions simultaneously: \(\text{Demand} = f(\text{location}, \text{time}, \text{covariates})\).

Motivating Example: iCloud Backup

iCloud Backup demand has clear spatio-temporal structure:

  • Temporal: Strong 24-hour cycles (peak at 2-4am local time), weekly cycles (more backups on weekends)
  • Spatial: Different peak times due to timezones; different demand levels by region
  • Interaction: Morning peak in Asia happens while Europe sleeps; this creates network load propagation

Spatio-Temporal Graph Neural Networks (ST-GNN)

Combines graph convolution (for space) with recurrent layers (for time):

# Pseudo-code: ST-GNN architecture # Input: time series for each region, graph of regions # Output: forecast for next time step For t in 1..T: 1. Temporal module: LSTM/GRU over time for each region h_t[region] = RNN(h_{t-1}[region], x_t[region]) 2. Spatial module: Graph convolution across regions h_spatial = GraphConv(h_t, adjacency_matrix) 3. Fusion: combine temporal and spatial prediction[t+1] = Dense(concat(h_t, h_spatial))

Why this works: The LSTM captures temporal dependencies within a region. The graph convolution lets regions "see" neighbors' patterns. Together, they learn that when Europe is in peak hours, nearby regions will soon follow.

ConvLSTM (Convolutional LSTM)

Treats spatial-temporal data as a grid (like pixels in video) and applies convolutional gates:

ConvLSTM Cell: $$i_t = \sigma(\mathbf{W}_{xi} * X_t + \mathbf{W}_{hi} * H_{t-1} + \mathbf{b}_i)$$ $$f_t = \sigma(\mathbf{W}_{xf} * X_t + \mathbf{W}_{hf} * H_{t-1} + \mathbf{b}_f)$$ $$(C_t, H_t) = \text{standard LSTM}$$

Where \(*\) denotes convolution. Good for 2D spatial grids (e.g., weather data). Less natural for irregular data center topologies.

Attention-Based Models

Use attention mechanisms to learn which regions' histories matter for forecasting a given region:

# Attention over spatial neighbors For forecasting region r: attention_weights = softmax( query[r] · keys[neighboring_regions] ) spatial_context = sum( attention_weights * values[neighboring_regions] ) forecast[r] = decoder(own_context[r], spatial_context)

Flexible and interpretable: you can inspect which regions' patterns the model learned to use.

Practical Simplified Approach for iCloud

Pragmatic Spatio-Temporal Strategy

Don't necessarily need fancy ST-GNN. Many successful forecasts use simpler models with good feature engineering:

1. Independent LSTM per region: Each region gets its own LSTM trained on its own historical demand.

2. Spatial features as covariates: Include lagged demand from neighboring regions as input features.

3. Hierarchical reconciliation: After independent forecasts, reconcile using MinT to enforce coherency.

Result: Captures temporal patterns locally, respects spatial coupling through reconciliation, much simpler than full ST-GNN.

Use Case: iCloud Backup Demand Forecast

Case Study: Predicting Overnight Backup Surge by Region

Goal: Forecast storage I/O (GB/sec) needed for backups in each region, 24 hours ahead.

Input data: Hourly backup I/O for each region, past 12 months. User count by region. Day-of-week, hour-of-day.

Approach:

  • Train separate LSTM for each region: past 168 hours (1 week) of I/O → next hour
  • Features: [hour_of_day, day_of_week, user_count_region, region_id, lagged_IO_1h/6h/24h/168h]
  • Include as covariates: lagged IO from neighbors (time zone shifted appropriately)
  • Ensemble: average LSTM forecast with autoregressive baseline

Reconciliation: Cluster-level forecasts sum to region, region-level sum to global. If sum of clusters < region forecast, allocate proportionally.

4.5 Vector Autoregression (VAR) and Cross-Dependencies

When multiple demand signals are highly correlated, should you model them jointly using Vector Autoregression (VAR)? Or keep them separate? This section answers that question.

VAR(p) Model

In VAR, each variable is a linear function of its own lags AND lags of all other variables:

VAR(p) System: $$\begin{bmatrix} y_{1,t} \\ y_{2,t} \\ \vdots \\ y_{K,t} \end{bmatrix} = \begin{bmatrix} c_1 \\ c_2 \\ \vdots \\ c_K \end{bmatrix} + \mathbf{A}_1 \begin{bmatrix} y_{1,t-1} \\ y_{2,t-1} \\ \vdots \\ y_{K,t-1} \end{bmatrix} + \cdots + \mathbf{A}_p \begin{bmatrix} y_{1,t-p} \\ y_{2,t-p} \\ \vdots \\ y_{K,t-p} \end{bmatrix} + \begin{bmatrix} \epsilon_{1,t} \\ \epsilon_{2,t} \\ \vdots \\ \epsilon_{K,t} \end{bmatrix}$$

Where \(K\) = number of series (e.g., CPU, memory, storage), \(p\) = lag order, \(\mathbf{A}_i\) = \(K \times K\) coefficient matrices.

Granger Causality

A key question in VAR: does knowing the history of series X help you forecast series Y (beyond just knowing Y's own history)?

Granger causality test: Fit two models:

Model 1: y_t = f(y_t-1, y_t-2, ...) [univariate] Model 2: y_t = f(y_t-1, y_t-2, ..., x_t-1, x_t-2, ...) [VAR] If Model 2's forecast error is significantly smaller, then x "Granger-causes" y.

Example in iCloud: Does storage demand history help forecast CPU demand? Probably not—they're driven by different services (backups vs request processing). So univariate CPU models are fine. But CPU and network bandwidth might be highly correlated; modeling jointly could help.

Dimensionality Curse: When VAR Breaks Down

VAR's problem: number of parameters grows quadratically in the number of series.

Number of Series (K) Lag Order (p) Parameters Practical Feasibility
3 (CPU, memory, storage) 4 3×(3×4) + 3 = 39 ✓ Easy
10 (multiple services) 4 10×(10×4) + 10 = 410 ✓ OK
50 (per-region, per-service) 4 50×(50×4) + 50 = 10,050 ⚠ Risky (overfitting)
1000+ (per-host demand) 4 1000×(1000×4) = 4,000,000+ ✗ Infeasible

When to Use VAR

VAR is Appropriate When:

✓ Small number of series (< 10-20)

✓ Strong cross-dependencies (e.g., CPU and network both driven by request volume)

✓ Medium-term forecasting horizon (where dynamics matter more than trends)

✓ Stable, stationary series (or differenced to be stationary)

VAR is Inappropriate When:

✗ Large number of series or you're forecasting 1000+ demand signals

✗ Weak cross-dependencies; series are largely independent

✗ Long-term forecasting; trend is more important than dynamics

✗ Highly non-stationary (trending) data; requires differencing

Better Alternatives to Full VAR

1. Univariate Models + Shared Covariates

Instead of VAR, forecast each series independently, but include common features:

# Shared covariates features = [ hour_of_day, day_of_week, user_count, requests_per_sec, anomaly_flags, ] # Forecast CPU using features cpu_forecast = model_cpu(cpu_history, features) # Forecast memory using same features memory_forecast = model_memory(memory_history, features) # Correlation comes through shared features, not explicit cross-lags

2. Factor Models

Assume all series are driven by a small number of latent factors. Estimate factors from data, then forecast factors and expand to individual series.

Reduces dimensionality dramatically while capturing correlation structure.

3. Hierarchical Reconciliation

Forecast CPU, memory, storage independently. Then use hierarchical reconciliation (MinT) to enforce consistency with total resource budget constraints.

Practical iCloud Scenario

CPU, Memory, and Network: VAR or Independent?

Situation: Forecasting CPU, memory, and network bandwidth for a cluster.

Dependencies:

  • CPU and network are strongly correlated (driven by request count)
  • Memory is semi-correlated (driven by session count, which lags request count)

Decision framework:

VAR(2) model: 3 series, 2 lags → 3×(3×2) + 3 = 21 parameters. Reasonable. Test if cross-lags matter (Granger test).

Independent + shared features: Include request_rate, user_count as covariates to all three models. Simpler, more interpretable.

Recommendation: Start with independent models + shared features. If residuals are significantly correlated, then upgrade to VAR(1) or VAR(2). Avoid VAR for more than 10-15 series.

5.2 Data Freshness and Handling Late-Arriving Data

In a distributed system like iCloud, telemetry from edge nodes may arrive late. A request processed on a device at 2:00 PM may not be logged to the central TSDB until 2:47 PM (or even days later for batch reconciliation). Forecasting models must handle this reality.

Late Arrival Sources

  • Network delays: Device → edge node → regional hub → central logging: 30-60 seconds typical
  • Batching: Edge nodes accumulate events, batch-send every 5 minutes: up to 5 min delay
  • Offline handling: Device offline when event occurs, syncs later when connected: hours to days
  • Batch reconciliation: End-of-day reconciliation jobs correct partial counts: delivered next day

Event Time vs Processing Time

Event time: When the event actually occurred (user initiated backup)

Processing time: When the event was recorded in the database

Critical rule for forecasting: Always model on event time, never processing time. If you use processing time, a late-arriving spike of old events will look like a sudden increase in demand, confusing your model.

Pitfall: Using Processing Time in Models

Scenario: Backup logs from devices in Asia arrive late due to network issues. When batch reconciliation runs, 500GB of backup data suddenly appears in the processing time stream on 2026-02-13, even though the backups happened on 2026-02-10.

If your model trains on processing time, it will learn that "demand increases on Tuesdays" (when batches arrive), not the true pattern.

Solution: Always have event_time and processing_time in your data. Filter/aggregate/train using event_time only.

Watermarking

A watermark is a timestamp indicating "as of this point, we have received all events with event_time up to X." Useful for knowing when data is "complete enough" to train.

# Watermark computation # For every day, compute: max(processing_time) - 24 hours # This threshold tracks when 95% of data for a given event_time arrives 2026-02-10 events: 95% arrived by 2026-02-11 10:00 (watermark) 2026-02-11 events: 95% arrived by 2026-02-12 10:00 (watermark) # When retraining models, only use data up to yesterday's watermark

Backfill and Idempotency

Backfill happens when historical data is recomputed or missing data is recovered. Your pipeline must handle backfills without duplicating data.

Pattern: Idempotent writes

# Each record has a unique ID based on (event_time, source, sequence_num) # Inserts use "UPSERT" (insert or update if ID already exists) # If the same backfill runs twice, second run overwrites first (no duplicate) INSERT INTO metrics (id, timestamp, value) VALUES (hash(timestamp || source || seq_num), timestamp, value) ON CONFLICT(id) DO UPDATE SET value = EXCLUDED.value;

Time Alignment

Ensure all features are aligned to the same time grid. If CPU data comes every 1 minute, but request counts come every 5 minutes, you need to align before feeding to a model.

# Align to common 5-minute grid cpu_1min.resample('5T').mean() # Downsample CPU to 5-min average requests_5min # Already 5-min granularity # Join on aligned timestamps features = cpu_1min.join(requests_5min, how='inner')

Timezone Handling

Apple's iCloud operates globally. Always store timestamps in UTC internally, but be aware of timezone effects in demand patterns.

# Store data timestamp_utc = datetime.utcnow() # When analyzing by "local hour" local_hour = (timestamp_utc + timedelta(hours=region_utc_offset)).hour # Example: 2026-02-12 08:00 UTC in Asia-Pacific (UTC+8) is local hour 16 (4pm)

5.3 Feature Stores

A feature store is a centralized repository for features: pre-computed, versioned, and served at training and inference time. It bridges the data lake and ML models.

Two-Layer Architecture

Layer Purpose Latency Freshness Typical Technology
Offline Batch computation for training data Hours (batch jobs) Updated daily BigQuery, Spark, Parquet files
Online Low-latency serving for real-time inference Milliseconds Updated on-demand or periodically Redis, DynamoDB, in-memory cache

Offline Feature Store

Batch-compute features from historical data, store as training datasets. Example: for each cluster on each day, compute rolling 7-day average CPU, peak CPU, variance, etc.

Point-in-time correctness: Critical! When training a model to forecast CPU on day D, all features must reflect what was known on day D-1 or earlier. No future data leakage.

# WRONG: includes day D's data when training to forecast day D SELECT cluster_id, date, avg(cpu) over (partition by cluster_id order by date rows between 6 preceding and current row) as cpu_roll_7d FROM metrics WHERE date BETWEEN '2026-01-01' AND '2026-02-12' # CORRECT: features computed from past data only SELECT f.cluster_id, f.date, m.cpu_roll_7d_avg # computed from dates < f.date FROM forecast_dates f LEFT JOIN ( SELECT cluster_id, date, avg(cpu) over (partition by cluster_id order by date rows between 6 preceding and 1 preceding) as cpu_roll_7d_avg FROM metrics ) m ON f.cluster_id = m.cluster_id AND f.date = m.date + 1

Online Feature Store

For real-time inference, you need features available in milliseconds. Typically use:

  • Redis: In-memory key-value store. O(1) lookup. Good for feature lookups by key (cluster_id).
  • DynamoDB: AWS managed key-value store. Eventual consistency, sub-100ms latency.
  • Elasticsearch: If you need aggregations/filters on feature values.

Pattern: Batch job runs every hour. Computes latest features from TSDB (last 7 days of data). Publishes to online store (Redis/DynamoDB). Serving code reads from online store on-demand.

# Batch update (hourly) SELECT cluster_id, avg(cpu) over (...) as cpu_roll_7d, max(cpu) over (...) as cpu_peak_7d FROM metrics WHERE date > current_date - 7 # Store in Redis with key: "features:{cluster_id}" # Value: JSON with all features # Inference-time lookup (milliseconds) features = redis.get(f"features:{cluster_id}") prediction = model.predict(features)

Feature Versioning and Lineage

Features change: you add a new feature, fix a bug in a feature definition, or change a parameter. You need to track what features were used for each model version.

Practice: Tag feature definitions with a version. Store lineage: which feature version was used for which model training run.

# Feature definition with version { "name": "cpu_roll_7d_avg", "version": "v2", # v1 had a bug in timezone handling "definition": "SELECT avg(cpu) over (PARTITION BY cluster ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)", "dependencies": ["metrics.cpu_util"], "last_updated": "2026-01-15" }

5.4 Model Monitoring and Drift Detection

A model in production drifts: input distribution changes, relationships between input and output change, or the underlying system behavior shifts. Detecting drift quickly enables fast retraining or rollback.

Data Drift vs Concept Drift

Type Definition Example Detection Method
Data Drift Distribution of input features changes New product launch increases user base by 30%, shifting traffic distribution Compare feature distributions (KL divergence, PSI)
Concept Drift Relationship between input and output changes (P(Y|X) changes) Users' backup behavior changes due to new device feature Compare residuals, forecast coverage, error metrics

Population Stability Index (PSI)

Measures how much a feature's distribution has shifted between training and production.

PSI Formula: $$\text{PSI} = \sum_{i=1}^{n} (p_i - q_i) \ln\left(\frac{p_i}{q_i}\right)$$

Where \(p_i\) = proportion of samples in bucket \(i\) during training, \(q_i\) = proportion in bucket \(i\) in production.

Interpretation:

  • PSI < 0.1: Negligible change, no action needed
  • PSI 0.1 - 0.25: Moderate shift, monitor but okay
  • PSI > 0.25: Significant shift, investigate and consider retraining

Example: CPU utilization in training was [5-30%]. In production (after new feature rollout), it's [15-45%]. Compute PSI on bucketed distribution.

Pitfall: PSI on Unbucketed Data

If you compute PSI on raw, continuous values, bins are arbitrary and PSI is sensitive to bucketing. Always bucket features before computing PSI, or use KL divergence on density estimates.

Residual Monitoring

Monitor forecast errors (residuals) in production. Systematic patterns indicate concept drift.

# Daily monitoring residuals = actual_cpu - forecast_cpu mean_residual = residuals.mean() if abs(mean_residual) > threshold: alert("Forecast bias detected") residual_std = residuals.std() if residual_std > train_std * 1.5: alert("Forecast uncertainty increased")

Coverage Monitoring for Quantile Forecasts

If your model produces quantile forecasts (p50, p95), check that coverage matches expectation.

# Model produces p95 forecast (95th percentile) # In production, it should contain ~95% of actual values coverage = (actual <= p95_forecast).mean() # Expected: ~0.95 # If coverage < 0.90: forecast is too narrow, under-provisioning # If coverage > 0.98: forecast is too wide, over-provisioning

Dashboard and Alerting

Key Monitoring Dashboards

Real-time Health:

  • Latest forecasts vs actuals (time series plot)
  • Forecast error distribution (histogram)
  • Model latency (serving time)

Drift Detection:

  • PSI trend over time (one line per feature)
  • Mean residual by cluster (heatmap)
  • Coverage for quantile forecasts

Alerts:

  • Mean absolute error > 15% for more than 2 hours
  • PSI > 0.25 for any top-5 features
  • P95 forecast coverage < 90%

5.5 Retraining Strategies and Deployment

Models degrade over time. Retraining brings models up-to-date with new data. But retraining can introduce new bugs or regressions. Smart retraining strategies balance freshness and stability.

Scheduled vs Trigger-Based Retraining

Strategy Frequency Pros Cons
Scheduled (Fixed) Daily, weekly, monthly Predictable, simple to implement, easy to debug May miss drift between retraining windows; may waste compute if no drift
Trigger-Based When drift detected Responsive, more efficient, captures sudden changes Complex to implement, needs reliable drift detector, can thrash if threshold is poorly tuned
Hybrid Scheduled + trigger Best of both: baseline schedule + emergency retrains More complex

Shadow Deployment

Deploy a new model alongside the old one. Both serve predictions, but only the old model's predictions are used in production. Compare them to validate the new model is good.

# Shadow deployment (48-72 hours typical) For each new forecast: old_model_prediction = production_model.predict(features) new_model_prediction = candidate_model.predict(features) actual = actuals.get() old_error = abs(old_model_prediction - actual) new_error = abs(new_model_prediction - actual) # Log for comparison log_metrics(old_error, new_error, candidate_id, timestamp) After 48 hours: # Compute aggregate metrics old_mae = mean(old_errors) new_mae = mean(new_errors) if new_mae < old_mae (or within threshold): promote candidate to production else: discard candidate, investigate

Canary Deployment

Roll out new model to a small percentage of traffic initially, monitor, then increase gradually.

# Canary deployment Day 1: 5% of clusters use new model Day 2: 25% if no issues Day 3: 50% if no issues Day 4: 100% if no issues If any day shows elevated error, stop and investigate

A/B Testing for Models

In some cases (e.g., recommendation models affecting user experience), you need statistical A/B testing to validate improvement.

# A/B test setup Control: Old model (Model A) Treatment: New model (Model B) Metric of interest: Forecast error for upcoming provision decisions Sample: 50% of clusters get model A, 50% get model B for 1 week # Statistical test error_A = model_a_errors error_B = model_b_errors t_stat, p_value = ttest_ind(error_A, error_B) if p_value < 0.05 and mean(error_B) < mean(error_A): Model B is significantly better, deploy fully

Automatic Rollback

If a new model degrades production metrics, automatically revert to the previous model.

# Automatic rollback criteria If in the last hour: - Mean absolute error > baseline + 20%, OR - P95 forecast coverage < 90%, OR - Model inference latency > 500ms Then: rollback to previous model trigger alert for investigation

6A.1 Rolling Aggregates (Window=N)

Rolling aggregates compute statistics over a moving window of time. Essential for capturing local trends and volatility in demand forecasting.

Pandas: Rolling Mean and Std

import pandas as pd import numpy as np # Sample data: cluster CPU utilization over time df = pd.DataFrame({ 'cluster_id': ['prod-west-01'] * 30, 'date': pd.date_range('2026-01-01', periods=30), 'cpu_util': np.random.uniform(20, 60, 30) # percentage }) # 7-day rolling mean and std, grouped by cluster df['cpu_roll_mean_7d'] = df.groupby('cluster_id')['cpu_util'].transform( lambda x: x.rolling(window=7, min_periods=1).mean() ) df['cpu_roll_std_7d'] = df.groupby('cluster_id')['cpu_util'].transform( lambda x: x.rolling(window=7, min_periods=1).std() ) print(df[['date', 'cpu_util', 'cpu_roll_mean_7d', 'cpu_roll_std_7d']])

Key parameters:

  • window: Number of periods to include. window=7 means 7 rows (days).
  • min_periods: Minimum observations required. min_periods=1 allows first row to have rolling_mean = itself. Use min_periods=window to avoid partial windows at edges.

PySpark Equivalent

from pyspark.sql import Window as SparkWindow from pyspark.sql.functions import avg, stddev, col # PySpark: Window function approach w = SparkWindow.partitionBy('cluster_id').orderBy('date').rowsBetween(-6, 0) # rowsBetween(-6, 0) includes current row and 6 rows before = 7 total df = df.withColumn('cpu_roll_mean_7d', avg('cpu_util').over(w)) df = df.withColumn('cpu_roll_std_7d', stddev('cpu_util').over(w)) df.select('date', 'cpu_util', 'cpu_roll_mean_7d', 'cpu_roll_std_7d').show()

Scaling considerations:

  • Pandas: Works on a single machine. Limited to available RAM. Good for clusters with < 10M rows of data.
  • PySpark: Distributed. Can handle billions of rows. Partition by cluster_id to parallelize rolling window computation.
Common Pitfall: min_periods in Rolling

Issue: If you don't set min_periods explicitly, the default is window=7. First 6 rows will have NaN, which breaks subsequent operations.

# WRONG df['roll_mean'] = df['cpu_util'].rolling(window=7).mean() # Result: NaN, NaN, NaN, NaN, NaN, NaN, 50.5 (first real value at index 6) # CORRECT df['roll_mean'] = df['cpu_util'].rolling(window=7, min_periods=1).mean() # Result: 48.2, 51.3, 49.8, 50.2, ... (all rows have a value)

When to use which: Use min_periods=1 if you want to handle edge cases. Use min_periods=window if you require full windows (discard edge data).

Multiple Windows

Compute rolling stats at multiple granularities (useful for capturing short and long-term trends):

# Multiple rolling windows for window_size in [3, 7, 14, 30]: df[f'cpu_roll_mean_{window_size}d'] = df.groupby('cluster_id')['cpu_util'].transform( lambda x: x.rolling(window=window_size, min_periods=1).mean() )

6A.2 Lag Features

Lags (previous values) are among the most predictive features in time series forecasting. CPU usage on day t-1 is highly predictive of day t.

Pandas: Creating Lags

# Create lag features for AR (autoregressive) model df = df.sort_values(['cluster_id', 'date']) # Single lag df['cpu_lag_1'] = df.groupby('cluster_id')['cpu_util'].shift(1) # Multiple lags for lag in [1, 7, 14, 28]: df[f'cpu_lag_{lag}'] = df.groupby('cluster_id')['cpu_util'].shift(lag) # Result: # date cpu_util cpu_lag_1 cpu_lag_7 cpu_lag_14 cpu_lag_28 # 2026-01-01 50.2 NaN NaN NaN NaN # 2026-01-02 51.5 50.2 NaN NaN NaN # 2026-01-08 52.1 49.8 50.2 NaN NaN # 2026-01-15 48.7 49.5 48.2 50.2 NaN # 2026-02-12 55.3 53.2 51.5 49.8 48.7

PySpark: Lag Function

from pyspark.sql.functions import lag w_lag = SparkWindow.partitionBy('cluster_id').orderBy('date') for lag_days in [1, 7, 14, 28]: df = df.withColumn( f'cpu_lag_{lag_days}', lag('cpu_util', lag_days).over(w_lag) )

Handling Missing Values After Lag

After creating lags, the first N rows have NaN. Several strategies:

# Strategy 1: Drop rows with NaN df_clean = df.dropna(subset=['cpu_lag_28']) # requires 28 days of history # Strategy 2: Forward fill (for status-like data) df['cpu_lag_7'] = df['cpu_lag_7'].fillna(method='ffill') # Strategy 3: Use interpolation (for continuous metrics) df['cpu_lag_7'] = df['cpu_lag_7'].interpolate(method='linear') # Strategy 4: Impute with global mean df['cpu_lag_7'] = df['cpu_lag_7'].fillna(df['cpu_util'].mean())
Which Lag Frequencies to Use

Day-of-week effect: Include lag_7 (previous week same day). CPU on Monday this week is similar to Monday last week.

Seasonal effect: Include lag_28 or lag_365 (4-week, annual seasonality).

Short-term momentum: Include lag_1 (previous day) and lag_2-3 (short-term trend).

Rule of thumb: Start with [1, 7, 14, 28]. Use feature importance to trim less useful lags.

(cumavg changes gradually, cummax monotonically increases)

6A.5 Handling Missing Timestamps and Gaps

Real-world data has gaps: a service is down, a cluster is taken offline, or data collection fails. You need to handle these gracefully.

Reindex to Fill Gaps

# Identify gaps: expected timestamps vs actual full_idx = pd.date_range( df['timestamp'].min(), df['timestamp'].max(), freq='1H' # expect 1-hour granularity ) df = df.set_index('timestamp').reindex(full_idx) # Result: rows for missing hours are added with NaN values # timestamp cluster_id cpu_util # 2026-01-01 00:00 prod-west-01 52.3 # 2026-01-01 01:00 NaN NaN <- GAP (no data) # 2026-01-01 02:00 prod-west-01 51.8

Strategy 1: Forward Fill (Carry Last Value Forward)

# Good for: step-like data (server status: UP/DOWN), categorical data # Assumption: value doesn't change until next measurement df['cpu_util'] = df['cpu_util'].fillna(method='ffill', limit=3) # limit=3: only fill up to 3 consecutive NaNs, then stop (don't fill gaps > 3 hours) # This assumes CPU stays same for up to 3 hours of missing data # (reasonable if we believe collection failure, not actual 0 demand)

Strategy 2: Interpolation (Continuous Data)

# Linear interpolation: assume value changes linearly over time df['cpu_util'] = df['cpu_util'].interpolate(method='linear') # Polynomial interpolation df['cpu_util'] = df['cpu_util'].interpolate(method='polynomial', order=2) # Time-based interpolation (uses index spacing) df['cpu_util'] = df['cpu_util'].interpolate(method='time')

Strategy 3: Drop Rows with Gaps

# Strict approach: drop any row with missing data # Only use if you have enough data and gaps are rare df_clean = df.dropna()
Pitfall: Interpolation Over Long Gaps

Issue: If you interpolate over a 24-hour gap using linear method, you're assuming the value changed smoothly. In reality, a 24-hour gap may indicate a genuine system issue.

Better: Use limit parameter to only interpolate short gaps.

# Fill gaps up to 3 hours, drop longer gaps df['cpu_util'] = df['cpu_util'].fillna(method='ffill', limit=3) df = df.dropna(subset=['cpu_util'])

6B.1 Window Functions Fundamentals

FROM daily_metrics WHERE date < CURRENT_DATE() -- Don't include today (no target yet)

6B.3 Rolling Sum and Average (ROWS BETWEEN)

Compute moving averages and cumulative sums using ROWS BETWEEN clause.

7-Day Rolling Average

-- 7-day rolling average CPU utilization SELECT cluster_id, date, cpu_util, AVG(cpu_util) OVER ( PARTITION BY cluster_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_7d_avg, STDDEV(cpu_util) OVER ( PARTITION BY cluster_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_7d_std FROM daily_metrics ORDER BY cluster_id, date

ROWS BETWEEN syntax:

  • ROWS BETWEEN 6 PRECEDING AND CURRENT ROW: includes 6 rows before + current = 7 total
  • ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW: all rows from partition start to current (cumulative)
  • ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING: current row ± 1 (centered window, good for smoothing)

Rolling Sum

-- Total request count over past 7 days (rolling sum) SELECT cluster_id, date, request_count, SUM(request_count) OVER ( PARTITION BY cluster_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_7d_request_total, -- Average requests per day over past 7 days AVG(request_count) OVER ( PARTITION BY cluster_id ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_7d_request_avg FROM daily_metrics
Pitfall: Off-by-One in ROWS BETWEEN

Common mistake: Using ROWS BETWEEN 7 PRECEDING AND CURRENT ROW when you want 7 days total.

-- WRONG: 8 rows total (7 preceding + current) AVG(...) OVER (... ROWS BETWEEN 7 PRECEDING AND CURRENT ROW) -- CORRECT: 7 rows total AVG(...) OVER (... ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)

Memory trick: If you want N rows, use (N-1) PRECEDING.

Multiple Granularities

Pitfall: Timezone-Aware Bucketing

Issue: TIMESTAMP_TRUNC uses UTC by default. If you bucket into "days", hour 23 UTC might be hour 15 (3pm) in a US timezone.

-- WRONG: buckets by UTC midnight TIMESTAMP_TRUNC(timestamp, DAY) -- CORRECT: convert to region timezone first TIMESTAMP_TRUNC( TIMESTAMP_ADD(timestamp, INTERVAL 8 HOUR), -- Convert to UTC+8 (Singapore) DAY )

Better approach: Store timezone_offset in your data, compute local hour explicitly.

6B.5 Point-in-Time Joins (Avoiding Data Leakage)

When training forecasting models, features must come from data available AT prediction time, not future data. This is point-in-time correctness.

The Problem: Data Leakage

-- WRONG: features include day D's data when predicting day D SELECT f.cluster_id, f.date AS forecast_date, m.cpu_util -- This is from the same day! Data leakage! FROM forecast_targets f LEFT JOIN metrics m ON f.cluster_id = m.cluster_id AND f.date = m.date

Correct: Point-in-Time Join

-- CORRECT: join to features computed from data BEFORE forecast date SELECT f.cluster_id, f.date AS forecast_date, -- Features computed from historical data only (date <= f.date - 1) m.cpu_roll_7d_avg, m.cpu_peak_7d, m.request_count_lag_1, m.request_count_lag_7 FROM forecast_targets f LEFT JOIN LATERAL ( -- Subquery: get latest feature snapshot before forecast date SELECT cpu_roll_7d_avg, cpu_peak_7d, request_count_lag_1, request_count_lag_7 FROM feature_snapshots fs WHERE fs.cluster_id = f.cluster_id AND fs.snapshot_date <= f.date - 1 -- Key: use past data only ORDER BY fs.snapshot_date DESC LIMIT 1 ) m ON TRUE

Intuition: When forecasting CPU for 2026-02-12, use features computed from data through 2026-02-11. This ensures features represent what was known AT prediction time.

Alternative: ASOF Join (DuckDB, Snowflake)

Topic 1: ML Foundations

Q1: Walk me through how you would choose between a gradient boosted tree model and a neural network for predicting CPU utilization across our data centers.

The choice depends primarily on your data characteristics and constraints. For tabular cloud telemetry data—which is what CPU utilization data typically is—gradient boosted trees (XGBoost, LightGBM) are the natural first choice: they handle mixed feature types well, require minimal feature engineering, and are highly interpretable through SHAP values. Neural networks shine when you have complex temporal dependencies across many time steps or when you need to capture nonlinear interactions in very high-dimensional feature spaces, but they require larger datasets, more careful tuning, and longer training times. For your data center prediction problem, I'd start with GBT because the dataset is likely modest in size, the features are mostly structured (time-of-day, resource requests, service metadata), and you need fast inference for real-time autoscaling decisions. Neural networks (RNNs or Transformer-based models like Temporal Fusion Transformers) become attractive only if you discover that your tabular features miss important temporal patterns that a larger training dataset could reveal, or if you need to forecast at extremely fine granularities with irregular sampling rates. The production serving latency is another critical factor: GBTs inference is sub-millisecond and deterministic, while neural nets require GPU resources or optimized batching infrastructure that adds operational complexity.

Follow-up: How would you handle high cardinality categorical features like service names?

For GBTs, I'd use target encoding (mean-encoding the service name by its average CPU impact), which reduces dimensionality while capturing the signal. In production, you'd store a lookup table to map each service to its encoded value, with a default for unseen services. Alternatively, embeddings work well: learn low-dimensional representations during training, then serve them via a simple lookup. If using neural networks, embeddings are built-in and more natural; the model learns service-specific patterns automatically.

Follow-up: What if the model needs to run in real-time for autoscaling decisions?

GBTs are ideal here: inference on a single sample runs in microseconds on CPU only. For autoscaling, you'd deploy the model in a lightweight service (Go or C++ wrapper) that consumes the latest metrics from your observability system and pushes predictions to the autoscaling controller. If using neural networks, you'd need GPU acceleration or heavy quantization/distillation to meet autoscaling latency SLOs, which is rarely justified for this use case.

Q2: Explain the quantile loss function and why it matters more than MSE for capacity planning.

MSE loss optimizes for the mean (conditional expectation), which is mathematically elegant but operationally risky for capacity planning: if your forecast is correct on average, you'll still face under-provisioning 50% of the time. Quantile loss (pinball loss) solves this by targeting a specific percentile. The quantile loss for quantile \(\tau\) is defined as \(L_\tau(y, \hat{y}) = \tau \max(y - \hat{y}, 0) + (1 - \tau) \max(\hat{y} - y, 0)\), which penalizes under-prediction (overage on the first term) \(\tau\) times more severely than over-prediction. For capacity planning, if you use \(\tau = 0.95\), the model learns to predict the 95th percentile of demand. This means your provisioning covers actual demand 95% of the time under normal conditions; the remaining 5% falls to auto-scaling buffers or graceful degradation. Using \(\tau = 0.99\) provides an even safer margin but costs more in idle capacity. The asymmetry in the loss function directly encodes your business constraint: over-provisioning is cheap (wasted compute), under-provisioning is catastrophic (service unavailability and customer impact).

Follow-up: How would you validate that your p95 forecast is actually calibrated?

Calibration means: when the model predicts the p95, actual demand exceeds that prediction approximately 5% of the time (no more, no less). You'd compute the empirical coverage on a held-out test set: count how many times actual demand \(y_i > \hat{y}_i^{0.95}\) and divide by the number of samples. If coverage is, say, 12%, your forecasts are under-calibrated (too optimistic), and you'd retrain or apply conformal prediction to recalibrate. A monthly dashboard tracking rolling 7-day coverage is standard practice.

Follow-up: What's the cost tradeoff between p90 and p99 provisioning?

p90 means 10% of spikes exceed capacity and need autoscaling or graceful degradation—higher operational risk but 5-10% lower infrastructure costs. p99 costs 20-30% more in idle capacity but guarantees coverage 99% of the time, which may be required for critical services. The optimal quantile depends on your SLA (e.g., availability target), cost of autoscaling bursts, and user tolerance for occasional slowdowns. I'd compute the cost curve: cost = \(\alpha \cdot \text{capacity} + \beta \cdot \text{unmet demand}\), then optimize for the right quantile.

Q3: A model has great training RMSE but poor test performance. Walk me through your debugging process.

This screams overfitting, and the diagnosis follows a structured path. First, I'd verify the train/test split: for time series data, you must never random-shuffle; instead, use forward-chaining (train on [t1, t2], test on [t3, t4], then train on [t1, t3], test on [t4, t5]). If you randomized a time series, the model has seen future information, which explains the gap. Second, I'd plot learning curves: if training loss continues decreasing while validation loss increases, it's classic overfitting—the model is memorizing training noise. Third, I'd scrutinize features for data leakage: are you using the target value itself, or a perfect proxy derived post-hoc? Fourth, I'd examine feature importance for suspicious features: if one feature dominates and has extremely high cardinality or low correlation with domain knowledge, it's a leakage signal. Fifth, I'd apply regularization: increase regularization strength (L1/L2), reduce tree depth (max_depth), use early stopping to halt training when validation loss plateaus, and maybe downsample features. Finally, I'd use proper time series cross-validation with rolling origin to measure performance realistically.

Follow-up: How would you detect data leakage specifically?

Build a feature importance plot and look for features with suspiciously high importance that don't make domain sense (e.g., a service name leaking information about future CPU). Check the correlation of each feature with the target: if a feature has near-perfect correlation and is computationally derived, it's likely a leak. In time series, look for features computed using the target variable (e.g., using tomorrow's request count to predict today's CPU). A simple test: remove the suspicious feature and retrain; if performance drops massively, it was leakage.

Follow-up: What's different about diagnosing overfitting in time series vs cross-sectional data?

Time series overfitting is harder to spot because random train/test splits can hide it entirely—the model may never see true future data. You must use time-aware evaluation (expanding or rolling windows) to surface the problem. Cross-sectional data can be randomly split with less risk because there's no temporal ordering, so even a random holdout set represents future distribution. For time series, temporal leakage is also more subtle: using today's observation to predict today's target is a classic mistake that random splits would catch but manual splits might miss.

Topic 2: Statistical Foundations

Q4: How do you test whether a time series is stationary, and what do you do if it's not?

Stationarity means the mean, variance, and autocovariance are constant over time—a critical assumption for many classical forecasting methods. I'd test it with two complementary statistical tests: the Augmented Dickey-Fuller (ADF) test, which has \(H_0: \text{series is non-stationary}\), and the KPSS test, which has \(H_0: \text{series is stationary}\). If ADF rejects the null (p-value < 0.05), the series is stationary; if KPSS fails to reject, it's also stationary. I'd also always plot the series visually: a clear trend or seasonal pattern is a visual red flag. If the series is non-stationary, I'd apply transformations: first differencing (using \(y_t - y_{t-1}\)) removes a linear trend, seasonal differencing (using \(y_t - y_{t-s}\) where \(s\) is the seasonal period) removes periodic components, and log transformation stabilizes variance when there's multiplicative seasonality. For iCloud storage demand, the series typically has a growth trend (non-stationary), so I'd difference it; CPU utilization often fluctuates around a stable mean (stationary), so I might not need transformation.

Follow-up: What if ADF and KPSS disagree?

This happens when the series is borderline (e.g., slow trend with high noise). Rely more on visual inspection and domain knowledge: if there's a visible trend or the series clearly doesn't return to the same level over months, treat it as non-stationary and difference it. When in doubt, differencing is safer—it doesn't hurt stationary data (you just get a differenced stationary series), but missing non-stationarity leads to spurious regressions.

Follow-up: Can you use non-stationary data directly in ML models like XGBoost?

Yes, and this is a key advantage of ML methods over classical econometrics. XGBoost and neural networks don't assume stationarity; they can learn trends and seasonality directly from features like time-of-day and day-of-year. However, you should still difference or detrend for classical ARIMA/SARIMA methods, which fundamentally require stationarity for the MLE estimator to work properly. For hybrid approaches (e.g., XGBoost on residuals from a detrended forecast), explicit differencing can improve performance.

Q5: Explain rolling-origin backtesting and why it's critical for time series model evaluation.

Rolling-origin backtesting is the gold standard for time series evaluation because it simulates how the model would perform in production. The procedure is: train on data from \(t=1\) to \(t=T_1\), generate predictions for \(t=T_1+1\) to \(t=T_1+h\) (where \(h\) is the forecast horizon), compute test error on that period, then roll forward: train on \(t=1\) to \(t=T_1+1\), predict \(t=T_1+2\) to \(t=T_1+h+1\), and repeat. This captures temporal variation in model accuracy—you'll discover if your model fails specifically during peak load, during rare events, or in certain seasons—which a random train/test split would hide. The alternative approaches are: expanding window (retrain on increasingly more data, better for non-stationary series with trend) and sliding window (retrain on fixed-size recent windows, better for capturing the most recent patterns). For capacity forecasting, I'd use expanding window with rolling-origin backtest, ensuring you evaluate across multiple complete seasonal cycles (include summer peaks, holiday periods, and quiet months) so you understand where the model struggles.

Follow-up: How do you decide between expanding window and sliding window?

Use expanding window if the relationship is slowly evolving but the long-term trend is predictable (e.g., storage demand with steady growth). Use sliding window if there are rapid regime changes or the model's assumptions have shifted dramatically (e.g., post-infrastructure migration, new service launches). You can also use expanding window with a minimum window size to avoid overfitting on tiny training sets early on. For iCloud, I'd lean expanding window with a warm start (e.g., minimum 12 months of data before first prediction).

Follow-up: How would you handle a regime change (like COVID) in your backtest?

Regime changes create distribution shift that the model hasn't seen. I'd flag them explicitly: mark pre-COVID and post-COVID periods separately, compute backtest metrics on each, and report them independently. If the model's performance degrades sharply during the regime change, acknowledge that limitation—the model wasn't trained on a similar shock. For forecasting during regime changes, use caution: either retrain on post-change data quickly, use weighted recent data (recent samples get higher weight), or build separate models for different regimes and switch between them based on leading indicators.

Q6: Design a forecasting system for iCloud storage demand that serves both autoscaling (minutes) and procurement (quarters).

This is a multi-horizon forecasting problem requiring a layered system because each horizon has different requirements, update frequencies, and feature sets. For the short-term layer (minutes to hours), I'd use a lightweight model—exponential smoothing or a shallow GBT—trained on high-frequency data (5-minute or hourly granularity) with very recent windows (last 7 days). This feeds the autoscaling controller directly, refreshing every 5 minutes; the model captures immediate demand patterns and burst spikes. For the medium-term layer (days to weeks), I'd train a gradient boosted tree with lag features (previous 7, 14, 28 days), calendar features (day-of-week, holidays), and event flags (product launches, marketing campaigns). Update this weekly and serve it for capacity planning discussions. For the long-term layer (months to quarters), I'd use trend extrapolation with scenario modeling: fit a trend component (e.g., exponential or polynomial growth), add seasonal adjustments, then layer in scenario analysis ("what if user base grows 30%?" or "what if data-per-user drops 10%?"). Long-term forecasts feed procurement and budget planning, updated monthly. Critically, use hierarchical reconciliation (e.g., MinT) to ensure that short-term forecasts, when aggregated, align with medium-term forecasts at the monthly level—this prevents the autoscaling layer from training data centers to a different demand profile than procurement procured for.

Follow-up: How would you handle an iPhone launch surge that's unprecedented in your training data?

Unprecedented events require intervention. Build a launch event model\(y_{\text{adjusted}} = y_{\text{predicted}} \times 1.3\). For procurement, engage with business teams early: they can flag upcoming product launches months in advance so you have time to provision capacity.

Follow-up: What's the cost of being wrong at each horizon?

Short-term: under-forecasting triggers expensive emergency autoscaling and brief latency spikes (high cost to user experience), over-forecasting costs a few minutes of wasted compute (low cost). Medium-term: errors compound; underforecasting requires costly mid-quarter infrastructure expansion, overforecasting locks in operational spend. Long-term: mistakes cascade into procurement decisions affecting 12-24 months of CAPEX budgets. Given these costs, short-term and medium-term should optimize for coverage (p90-p95), while long-term should be conservative with scenario analysis to hedge risk.

Q7: Explain conformal prediction and why it's valuable for capacity planning.

Conformal prediction is a distribution-free framework that constructs prediction intervals with finite-sample coverage guarantees. The key idea is deceptively simple: compute a nonconformity score on a calibration set (a holdout portion of data) to quantify how surprising each prediction is, then use the empirical quantile of these scores to construct prediction intervals. Concretely, if \(\alpha = 0.05\) (you want 95% coverage), you take the \(\lceil (n+1)(1-\alpha) \rceil\)-th order statistic of the nonconformity scores as your threshold. A new prediction interval is guaranteed to contain the true value with at least \(1 - \alpha\) probability, regardless of the underlying distribution or model—no normality assumptions needed. This is revolutionary for capacity planning because cloud demand is often multimodal, heavy-tailed, and heteroscedastic (variance changes with time of day), which violates standard parametric assumptions. You can wrap any model (XGBoost, neural nets, even simple linear regression) with conformal prediction and get calibrated intervals automatically. Practically, you split your data into train, calibration, and test: train the model on train, compute nonconformity on calibration, and evaluate coverage and interval width on test. This gives you prediction intervals you can actually trust for capacity decisions.

Follow-up: What assumptions does conformal prediction make?

The core assumption is exchangeability: the calibration and test sets come from the same distribution, and their ordering is irrelevant. This is satisfied for i.i.d. data (easy) but violated for time series where future data often differs from the past (harder). Adaptive conformal prediction and other variants address time series by using weighted calibration sets that prioritize recent data. The method also assumes the nonconformity score is a good measure of surprise—choosing the right score (e.g., absolute error, quantile loss) matters.

Follow-up: How do you handle non-exchangeable data like time series in conformal prediction?

Use conformal quantile regression with online updating: maintain a rolling window of nonconformity scores from recent test periods, and update your threshold incrementally. Or use a "weighted" version where older calibration samples get lower weight. Another approach: split your time series into contiguous folds (e.g., train on 2021, calibrate on 2022, test on 2023), ensuring the calibration and test sets are temporally separated from training—this relaxes exchangeability to a weaker "stationarity" assumption.

Q8: You notice your p95 forecast is only covering 88% of actuals. What's your diagnosis and fix?

This is a critical under-calibration problem—the forecast is 7 percentage points too optimistic, meaning 12% of the time demand exceeds your p95 prediction. This directly translates to provisioning shortfalls that trigger cascading failures, so diagnosis is urgent. I'd investigate in this order: (1) Concept drift—has the relationship between features and demand changed? Compute Population Stability Index (PSI) on key features to see if their distributions have shifted. A PSI > 0.1 indicates significant drift. (2) Training issue—was the quantile loss properly optimized during training? Check that the loss function used \(\tau = 0.95\) consistently, not accidentally reverted to MSE. (3) Underrepresentation of extreme events—if your training data has only mild demand spikes but recent actuals include unprecedented spikes, the model hasn't learned the tail well. (4) Aggregation mismatch—are you comparing predictions at one aggregation level (e.g., cluster) but forecasting at another (e.g., data center)? The reconciliation may have corrupted the percentile. Once diagnosed, the immediate fix is recalibration using isotonic regression: fit a monotone function on recent test data to map your original predictions to properly calibrated ones, without retraining. Or apply conformal prediction on recent data to recompute your threshold. As a temporary safety valve, increase your quantile target from p95 to p97 while investigating root cause. Set up weekly coverage monitoring on a dashboard to catch recurrence early.

Follow-up: How would you communicate this risk to stakeholders?

Be direct: "Our capacity forecast is underestimating demand 12% of the time, creating outage risk. We're investigating the root cause and applying a short-term fix (p97 instead of p95) that costs ~2% more compute. Resolution in 2 weeks." Provide a quantified impact: "This means approximately 10-20 hours per month of potential capacity shortfalls if unaddressed." Get buy-in from engineering and finance before making changes to procurement or autoscaling behavior.

Follow-up: What's the difference between recalibration and retraining?

Recalibration is lightweight: take your existing model's predictions and post-process them (e.g., via isotonic regression) to adjust for bias, without touching model weights. This is fast (minutes) and safe. Retraining rebuilds the entire model from fresh data, which can fix deeper issues (new features, structural changes) but takes hours and risks introducing new problems. For drift issues, recalibration is first aid; retraining is the cure. Use recalibration when drift is temporary, retraining when it's structural.

Topic 4: Hierarchical Forecasting

Q9: How would you ensure that your forecasts at the cluster level, data center level, and region level are all consistent?

This is the hierarchical reconciliation problem: forecasts generated independently at different levels of aggregation will violate the accounting identity that cluster-level forecasts must sum to data center forecasts, which sum to region forecasts. The solution is MinT (Minimum Trace) reconciliation, which works as follows. First, generate base forecasts independently at all levels (using your favorite models—GBTs, ARIMA, whatever). Second, encode the hierarchy as a summation matrix \(S\) that specifies which clusters roll up to which data centers, etc. Third, estimate the covariance matrix \(\Sigma\) of the base forecast errors (on a holdout validation set). Finally, compute the optimal combination of base forecasts: \(\hat{y}_{\text{reconciled}} = (S (S^T \Sigma^{-1} S)^{-1} S^T \Sigma^{-1}) \hat{y}_{\text{base}}\), which ensures coherency (summing constraint) while minimizing total forecast variance. The intuition: forecasts with low error variance are weighted more heavily. In practice, estimating \(\Sigma\) is high-dimensional; you'd use shrinkage estimators or OLS constraints to regularize. An alternative is bottom-up forecasting: predict only at the cluster level (bottom of the hierarchy), then sum up. This is simpler but noisier because cluster-level forecasts are harder than aggregate forecasts. For iCloud capacity, I'd use MinT with monthly reconciliation: update covariance estimates monthly, recompute reconciliation weights, feed reconciled forecasts to procurement systems that require consistency across all levels.

Follow-up: What are the computational challenges of MinT at scale?

With hundreds of data centers and thousands of clusters, the summation matrix \(S\) and covariance \(\Sigma\) become huge. Inverting \((S^T \Sigma^{-1} S)\) is \(O(n^3)\), which is infeasible. Solutions: (1) Block-wise reconciliation—reconcile each region independently, (2) Regularized covariance estimation to make the matrix well-conditioned, (3) Approximate methods like iterative refinement instead of exact inversion. I'd start with block-wise MinT (reconcile each region separately), which reduces dimension by 100x or more.

Follow-up: When would you prefer bottom-up over MinT?

Bottom-up is preferable when base forecasts at the top level (region) are structurally biased or when cluster-level forecasts are much more accurate than aggregate forecasts. It's also simpler to implement and explain. But it ignores covariance information and can amplify cluster-level noise. Use bottom-up when you trust the bottom-level model quality, MinT when base forecasts at multiple levels are equally good and you want to extract maximum information.

Q10: How would you forecast demand when there are strong correlations between different services?

Correlated services create dependencies: if one service spikes, others may follow. First, I'd quantify the correlations with cross-correlation analysis: compute lagged correlations to see if service A leads service B by hours or days (e.g., compute correlation at lag -4, 0, +4 hours). If correlations are strong and the number of services is small (<20), Vector AutoRegression (VAR) is natural: it models the joint dynamics of all series, where each series is a linear function of its own lags and the lags of all other series. However, VAR explodes in dimension: with 1000 services, the parameter matrix becomes \(1000 \times 1000 \times \text{lags}\), which is impractical to estimate stably. For large-scale cloud systems, a better approach is feature-based coupling: build independent forecasting models per service, but include lagged values of correlated services as features. This captures cross-service dynamics without parameter explosion. Or use a neural architecture with shared embeddings: embed each service into a low-dimensional space, then use an attention mechanism (like Temporal Fusion Transformer) to learn which other services matter for each prediction. For final reconciliation, apply hierarchical constraints (if Service A depends on Service B, their forecasts must be consistent at the aggregate level). The practical approach for iCloud: detect correlated service pairs via correlation analysis, include those pairs as cross-features in GBT models, and validate that predicted totals match hierarchical reconciliation targets.

Follow-up: Give an example of correlated iCloud services.

Photo storage and iCloud Drive are highly correlated: when users upload photos, they often also sync documents and files. Email and contacts are bundled, so mail storage growth correlates with contact sync demand. If a user upgrades their iCloud plan, all storage services increase together. These natural correlations should be captured via cross-features or explicit covariance modeling.

Follow-up: How would you detect if a correlation has broken (structural break)?

Compute rolling correlations over a 30-day window and look for sudden drops or reversals. A change-point detection algorithm (PELT, binary segmentation) can identify when the correlation coefficient shifts significantly. If a correlation breaks—say, Photo and Drive used to correlate at 0.8 but drop to 0.2—this signals a structural change (e.g., new pricing tier unbundled services, or a new service launched independently). Retrain your feature-based models to drop the stale cross-features and re-learn dynamics on post-break data.

Topic 5: Infrastructure & Operations

Q11: How would you design a feature store for a real-time capacity forecasting system?

A feature store is a centralized repository for features that decouples feature engineering from model training and serving, preventing duplicated logic and keeping features in sync. I'd design a two-tier architecture. The offline store (e.g., BigQuery, Hive, S3) handles batch-computed features updated on a schedule (e.g., daily): 7-day rolling averages of CPU utilization, trend components extracted from historical demand, seasonal indices learned from past years, and aggregate statistics (p50, p95) per cluster. These feed training pipelines and medium/long-term forecasting models. The online store (e.g., Redis, DynamoDB, Cassandra) serves real-time features with sub-millisecond latency: the last 30 minutes of CPU readings, current request rate, recent lag values (CPU from 1 hour ago, 24 hours ago). This feeds autoscaling models. Point-in-time correctness is critical: when training a model to predict demand at time \(t\), features must reflect only what was available at time \(t\)—no future information leakage. Implement versioning: track which feature definitions and source data were used at each point in time. Use a backfill framework to compute historical features consistently; this is often the hardest part. For iCloud, offline features come from a daily batch job (Spark, Airflow), while online features stream from your observability system (Prometheus, Datadog) into Redis with 5-second update latency. The forecasting service queries the online store and the offline store, assembles the feature vector, and serves predictions to autoscaling or capacity planning systems.

Follow-up: How do you ensure point-in-time correctness during model training?

Use a time-travel API in your feature store: given a prediction timestamp, the API returns the features as they existed at that moment—not including any data that arrived later. Concretely, tag each feature row with a created_at timestamp, and filter WHERE created_at <= prediction_time when building training sets. For streaming features, store them immutably (no updates, only inserts) with timestamps. Many feature stores (Tecton, Feast) provide this built-in; if building custom, careful data pipeline design is essential.

Follow-up: What happens if the online store has stale data?

Stale features degrade model accuracy and can trigger poor autoscaling decisions. Monitor freshness: set SLOs like "90% of features refreshed within 5 seconds." If staleness is detected (e.g., a feature hasn't been updated for >10 seconds), either fallback to a default value or trigger a fallback model that doesn't depend on that feature. Alert on persistent staleness so the data engineering team investigates the upstream data pipeline.

Q12: Your forecasting model's error has increased 30% over the past month. Walk me through your investigation.

A 30% increase in error is significant and requires systematic debugging. My investigation follows this order: (1) Data quality issues—is the input data pipeline broken, delayed, or missing? Check for NaNs, outliers, schema changes, or gaps in the time series. Query your observability system directly (Prometheus, CloudWatch) to verify that data flowing into the feature store matches production reality. If the pipeline is corrupt, fix it immediately and retrain. (2) Feature distribution shift—compute Population Stability Index (PSI) on each feature: \(\text{PSI} = \sum (\text{actual}_i - \text{expected}_i) \log(\text{actual}_i / \text{expected}_i)\). If PSI > 0.1 for any feature, that feature's distribution has drifted out-of-distribution and the model is extrapolating blindly. (3) Concept drift—has the relationship between features and target changed? Compare feature importances from before and after the error increase; if they've shifted dramatically, the model's decision boundaries no longer match reality. This might signal a new service launch, infrastructure migration, or changed user behavior. (4) Seasonality or cyclical patterns—is this error increase seasonal (e.g., first major holiday event the model hasn't seen), or is it a one-time regime shift? (5) Error segmentation—break down error by cluster, service, time-of-day, and day-of-week. Concentrated error (one cluster degraded) suggests a local issue; uniform degradation suggests global drift. Once diagnosed, remediate: data quality → fix pipeline, feature shift → reweight recent data or trigger retraining, concept drift → retrain on new data, seasonal unseen event → manually adjust predictions or flag for monitoring.

Follow-up: How would you automate this investigation?

Build a model monitoring dashboard that tracks: (1) prediction error (MAE, RMSE) segmented by cluster/region/service, (2) PSI for each feature with alerts at PSI > 0.1, (3) feature importance and comparison to baseline, (4) data freshness and completeness, (5) inference latency. On error spike, the dashboard auto-computes these diagnostics and generates a ranked list of likely causes. Set up thresholds: if error increases 20% and PSI increases on 3+ features, the system automatically flags "feature drift detected" and recommends retraining. This reduces manual investigation time from hours to minutes.

Follow-up: How do you decide between retraining and rebuilding the model from scratch?

Retrain (keeping the same architecture, just fitting new weights) when the drift is gradual and features remain informative—this is 95% of cases. Rebuild from scratch when: (1) the error increase is due to a fundamental change (e.g., metrics collection changed format, new service dominates demand), (2) feature importances have completely reshuffled, or (3) the original model architecture (e.g., GBT tree depth) is now inappropriate. In practice, try retraining first (takes hours), monitor performance, and escalate to rebuild if retraining doesn't recover accuracy.

Topic 6: Python & SQL

Q13: Write SQL to find the top-3 clusters by p95 CPU utilization for each region over the past 7 days.

Here's the SQL query:

WITH cluster_p95 AS ( SELECT region, cluster_id, APPROX_QUANTILES(cpu_util, 100)[OFFSET(95)] AS p95_cpu FROM hourly_metrics WHERE date >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY) GROUP BY region, cluster_id ), ranked AS ( SELECT *, ROW_NUMBER() OVER (PARTITION BY region ORDER BY p95_cpu DESC) AS rn FROM cluster_p95 ) SELECT region, cluster_id, p95_cpu FROM ranked WHERE rn <= 3 ORDER BY region, rn;

Explanation: The first CTE cluster_p95 groups by region and cluster, then computes the 95th percentile using APPROX_QUANTILES() (BigQuery syntax; adjust for your database). Follow-up: How do you prevent leakage when computing rolling features?

Leakage occurs if you compute rolling stats using future data. To prevent it, ensure the rolling window only uses past data: x.rolling(window, min_periods=1).mean() by default rolls backward (includes current row and past), which is correct for forecasting. Never use forward-looking operations like x.rolling(window).mean().shift(-window), which is wrong. When splitting train/test, ensure the test set starts after the lag window (e.g., train on days 1-300, don't compute features for days 1-28, test on days 301+ where features have full lag history). In production, be extra careful: when predicting for time \(t\), your rolling features must use only data up to time \(t-1\), not \(t\) itself.

Q15: Write SQL to compute week-over-week growth rate for each service, flagging any service with >20% growth as a capacity risk.

WITH weekly AS ( SELECT service_id, DATE_TRUNC(date, WEEK) AS week_start, SUM(request_count) AS weekly_requests FROM service_metrics WHERE date >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY) GROUP BY service_id, week_start ), with_growth AS ( SELECT service_id, week_start, weekly_requests, LAG(weekly_requests) OVER ( PARTITION BY service_id ORDER BY week_start ) AS prev_week, SAFE_DIVIDE( weekly_requests - LAG(weekly_requests) OVER ( PARTITION BY service_id ORDER BY week_start), LAG(weekly_requests) OVER ( PARTITION BY service_id ORDER BY week_start) ) AS wow_growth FROM weekly ) SELECT service_id, week_start, weekly_requests, prev_week, ROUND(wow_growth * 100, 2) AS wow_growth_pct, CASE WHEN wow_growth > 0.20 THEN 'CAPACITY_RISK' WHEN wow_growth < -0.20 THEN 'CAPACITY_DECLINE' ELSE 'NORMAL' END AS status FROM with_growth WHERE week_start = (SELECT MAX(week_start) FROM weekly) AND wow_growth IS NOT NULL ORDER BY wow_growth DESC;

Explanation: The first CTE weekly aggregates request counts to the weekly level using DATE_TRUNC(date, WEEK) (BigQuery; use DATE_TRUNC('week', date) in PostgreSQL). The second CTE with_growth computes lag features: LAG(...) OVER (...) retrieves the previous week's value for each service. SAFE_DIVIDE prevents division by zero (returns NULL if denominator is 0, rather than erroring). The final SELECT filters for the latest week only and displays growth rate. The status column flags >20% growth as a capacity risk (demand surging, may need immediate provisioning) and <-20% as a decline (potential over-provisioning). ROUND(wow_growth * 100, 2) converts to percentage with 2 decimals for readability.

Tip: The WHERE date >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY) keeps the query efficient by scanning only recent data, avoiding full table scans. For very large tables, further optimize with table partitioning on date.
