SECTION 1

Introduction & General Knowledge

Capacity planning at hyperscale is an optimization problem. You have finite budget, finite data-center power, and finite hardware supply. The question is: what to buy, where to place it, and how to assign workloads — while minimizing cost and meeting all service-level objectives. LP, MIP, and Bin Packing are the three primary formulation families used to solve this.

LP — Linear Programming Fundamentals

An LP is an optimization problem where the objective function and all constraints are linear, and all decision variables are continuous (can take any real value).

Standard LP Form $$\min \; \mathbf{c}^\top \mathbf{x}$$ $$\text{subject to:} \quad A\mathbf{x} \geq \mathbf{b}, \quad \mathbf{x} \geq \mathbf{0}$$
SymbolTypeMeaning
\(\mathbf{x} \in \mathbb{R}^n\)Decision vectorThe unknowns we are solving for (e.g., amount of each resource to allocate)
\(\mathbf{c} \in \mathbb{R}^n\)Cost vectorPer-unit cost coefficient for each decision variable
\(A \in \mathbb{R}^{m \times n}\)Constraint matrixEncodes how each variable contributes to each constraint
\(\mathbf{b} \in \mathbb{R}^m\)RHS vectorRight-hand side: the minimum (or maximum) requirements
Key LP Properties
  • Solvable in polynomial time — interior-point methods are O(n³·L); simplex is exponential worst-case but fast in practice.
  • Optimal solution lies at a vertex of the feasible polyhedron (simplex exploits this).
  • Strong duality — the dual problem gives exact shadow prices for every constraint.
  • No integrality — you can buy "3.7 servers," which is unrealistic for hardware procurement but fine for continuous resource allocation (CPU shares, bandwidth fractions).

MIP — Mixed-Integer Programming Fundamentals

A MIP extends LP by requiring some decision variables to be integers. When all integer variables are binary (0 or 1), it's a Binary Integer Program (BIP). When all variables are integer, it's a pure Integer Program (IP).

Standard MIP Form $$\min \; \mathbf{c}^\top \mathbf{x} + \mathbf{d}^\top \mathbf{y}$$ $$\text{subject to:} \quad A\mathbf{x} + B\mathbf{y} \geq \mathbf{b}$$ $$\mathbf{x} \geq \mathbf{0}, \quad \mathbf{y} \in \mathbb{Z}_{\geq 0}$$
SymbolTypeMeaning
\(\mathbf{x} \in \mathbb{R}^{n_1}\)Continuous variablesFractional quantities (e.g., CPU shares allocated, bandwidth fraction)
\(\mathbf{y} \in \mathbb{Z}^{n_2}\)Integer variablesWhole-unit quantities (e.g., number of servers to purchase)
\(\mathbf{c}, \mathbf{d}\)Cost vectorsPer-unit cost for continuous and integer variables respectively
\(A, B\)Constraint matricesCoefficients linking variables to constraints
\(\mathbf{b}\)RHS vectorResource demands, budget caps, capacity limits
MIP Complexity: NP-Hard

Adding integrality constraints makes the problem NP-hard. Solvers use branch-and-bound (systematically explore integer assignments), cutting planes (tighten the LP relaxation), and heuristics (find good feasible solutions early). For capacity planning problems with 10K–100K integer variables, modern solvers (Gurobi, CPLEX) typically converge within minutes to hours, depending on problem structure and the acceptable MIP gap.

Bin Packing — Fundamentals

The Bin Packing Problem (BPP) asks: given a set of items (services/workloads) each with a known size, and a set of bins (servers/nodes) each with a known capacity, assign all items to bins such that no bin exceeds its capacity, while minimizing the number of bins used.

Classic 1-D Bin Packing $$\min \sum_{j=1}^{m} z_j$$ $$\text{s.t.} \quad \sum_{i=1}^{n} w_i \cdot x_{ij} \leq C \cdot z_j \quad \forall \; j = 1, \ldots, m$$ $$\sum_{j=1}^{m} x_{ij} = 1 \quad \forall \; i = 1, \ldots, n$$ $$x_{ij} \in \{0,1\}, \quad z_j \in \{0,1\}$$
SymbolTypeMeaning
\(n\)ParameterNumber of items (service instances to place)
\(m\)ParameterNumber of available bins (servers); set as upper bound \(m = n\)
\(w_i\)ParameterSize of item \(i\) (e.g., CPU cores required by service \(i\))
\(C\)ParameterCapacity of each bin (e.g., total CPU cores per server)
\(x_{ij} \in \{0,1\}\)Decision variable1 if item \(i\) is assigned to bin \(j\), 0 otherwise
\(z_j \in \{0,1\}\)Decision variable1 if bin \(j\) is used (has at least one item), 0 otherwise

In capacity planning, multi-dimensional bin packing is more realistic: each item has requirements in multiple dimensions (CPU, RAM, GPU, storage, network), and each bin has capacities in all dimensions. This is also called vector bin packing (VBP).

Typical Pipeline: LP → MIP → Bin Packing

In practice, capacity planning uses all three in a pipeline:

  1. LP relaxation first — solve quickly to get shadow prices, feasibility check, and a lower bound on cost.
  2. MIP for procurement — using the LP solution as a warm start, solve with integrality to get the exact number of each node type per data center.
  3. Bin packing for placement — given the purchased fleet, assign individual service instances to specific machines in real-time.

Duality & Shadow Prices

Every LP has a dual problem. Where the primal asks "minimize cost subject to meeting demand," the dual asks "maximize the value of demand subject to not exceeding unit costs." The fundamental theorem:

Strong Duality Theorem $$\text{If the primal LP has an optimal solution } \mathbf{x}^*, \text{ then the dual has an optimal solution } \boldsymbol{\lambda}^* \text{ with:}$$ $$\mathbf{c}^\top \mathbf{x}^* = \mathbf{b}^\top \boldsymbol{\lambda}^*$$
SymbolNameInterpretation in Capacity Planning
\(\lambda_i^*\)Shadow price (dual variable) for constraint \(i\)Marginal cost reduction if the RHS of constraint \(i\) is relaxed by one unit. E.g., "if we had one more GPU, total cost would decrease by \(\lambda_{\text{GPU}}^*\) dollars."
\(\lambda_i^* = 0\)Non-binding constraintThis resource has slack — we have more than enough. Buying more provides zero benefit.
\(\lambda_i^* > 0\)Binding constraintThis resource is at its limit. Every additional unit is worth \(\lambda_i^*\) in cost savings.
Shadow Prices in MIP

MIP problems do not have exact duals because of the integrality requirement. However, the LP relaxation (drop integrality) gives approximate shadow prices. These are still extremely useful: they indicate which constraints are tight and what the marginal value of relaxation would be if we could buy fractional servers. In practice, LP relaxation shadow prices are the primary tool for resource valuation in capacity planning.

Binding Constraints

A constraint is binding (or active) at the optimal solution if it holds with equality. A constraint is non-binding (has slack) if there is room to spare.

Binding: \(a_i^\top \mathbf{x}^* = b_i\)

The resource is fully consumed. Shadow price \(\lambda_i^* > 0\). This is the bottleneck — adding more of this resource directly improves the objective.

Example: GPU capacity constraint is binding — you've used every GPU you have. The shadow price tells you what each additional GPU is worth.

Non-binding: \(a_i^\top \mathbf{x}^* > b_i\)

The resource has slack: \(s_i = a_i^\top \mathbf{x}^* - b_i > 0\). Shadow price \(\lambda_i^* = 0\). Buying more of this resource does nothing.

Example: CPU constraint is non-binding with 2,000 cores of slack — CPUs are abundant because you over-purchased to satisfy memory demand.

Complementary Slackness

A fundamental LP result: at optimality, \(\lambda_i^* \cdot s_i = 0\) for every constraint. Either the constraint is tight (slack = 0, shadow price possibly > 0) or it has slack (shadow price = 0). Both cannot be positive simultaneously. This is the mathematical reason shadow prices work as "scarce-resource indicators."

Penalties & Soft Constraints

Some constraints in capacity planning are "soft" — violating them is undesirable but not physically impossible. We model soft constraints using penalty variables and penalty costs in the objective function.

Hard vs. Soft Constraint Conversion

Hard constraint (demand must be met exactly):

$$\sum_{n} a_{r,n} \cdot x_n \geq D_r$$

Soft constraint (demand can be unmet, at a cost):

$$\sum_{n} a_{r,n} \cdot x_n + u_r \geq D_r, \quad u_r \geq 0$$ $$\text{Add to objective:} \quad + \; M_r \cdot u_r$$
SymbolTypeMeaning
\(u_r \geq 0\)Slack / penalty variableAmount of demand \(r\) left unmet
\(M_r\)Penalty coefficientCost per unit of unmet demand. Set very high (Big-M) to make unmet demand a last resort; set moderately to allow graceful shedding of low-priority demand.
Service Tiering via Penalties

Different services get different penalty costs: Tier-0 (core iCloud auth): \(M = \$100{,}000\) per unmet unit — effectively a hard constraint. Tier-1 (Photos sync): \(M = \$10{,}000\). Tier-2 (analytics batch jobs): \(M = \$500\). Under budget pressure, the optimizer will shed Tier-2 demand first, then Tier-1, and never Tier-0.

MIP Gap

MIP solvers don't always find the proven optimal solution within a practical time limit. The MIP gap measures how close the best found solution is to the theoretical best possible.

MIP Gap Definition $$\text{MIP Gap} = \frac{z_{\text{incumbent}} - z_{\text{LP bound}}}{z_{\text{incumbent}}} \times 100\%$$
SymbolMeaning
\(z_{\text{incumbent}}\)Objective value of the best feasible integer solution found so far. This is an upper bound (for minimization).
\(z_{\text{LP bound}}\)Objective value of the LP relaxation (continuous relaxation of the MIP). This is a lower bound — no integer solution can be cheaper than the LP relaxation.
MIP Gap = 0%Proven optimal: the incumbent equals the bound.
MIP Gap = 2%The best solution found is within 2% of the best theoretically possible. Often acceptable for practical planning.
Practical MIP Gap Targets
Planning ContextAcceptable GapRationale
Strategic capacity plan (millions of $)0.5–1%High stakes; worth waiting for near-optimal
Quarterly procurement cycle1–3%Moderate stakes; solve must complete in minutes
Daily rebalancing / scheduling5–10%Frequent re-solve; speed matters more than precision
Real-time cluster schedulingHeuristic onlySub-second latency; use greedy bin packing

But if DC-East were also binding, the shadow price would jump dramatically.

MIP Formulation — Detailed Example

A procurement model: decide how many servers of each type to buy for each data center and how to assign service workloads to them, minimizing total cost while satisfying all demand and budget constraints. The key difference from LP: you must buy whole servers.

Scenario Setup

Problem Data

Node types (catalog):

Node type \(n\)CPU cores \(a_n^{\text{cpu}}\)RAM GB \(a_n^{\text{ram}}\)GPUs \(a_n^{\text{gpu}}\)Power kW \(P_n\)Monthly cost \(c_n\)
Compute-L1921,02400.7$870
Memory-XL962,04800.6$1,050
GPU-Train1282,048810.0$6,200
GPU-Infer6451242.5$1,350

Service demands (same as LP scenario): Photos API (10K cores, 40K GB), ML Inference (2K cores, 16K GB, 500 GPUs), KV Cache (4K cores, 80K GB).

Additional constraints: Budget cap \(B = \$500{,}000\)/month. DC-West power limit \(W_1 = 2{,}000\) kW. DC-East power limit \(W_2 = 1{,}500\) kW.

Complete Variable Dictionary

VariableDomainMeaningUnits

\(y_{s,d} \in \mathbb{R}_{\geq 0}\)Non-negative realFraction (or count) of service \(s\) demand served from DC \(d\)Fraction of total demand
\(u_s \in \mathbb{R}_{\geq 0}\)Non-negative realUnmet demand for service \(s\) (penalty variable)Same units as demand

Parameters:

ParameterMeaning
\(\mathcal{N}\) = {Compute-L, Memory-XL, GPU-Train, GPU-Infer}Set of node types in the catalog
\(\mathcal{D}\) = {DC-West, DC-East}Set of data centers
\(\mathcal{S}\) = {Photos, ML-Infer, KV-Cache}Set of services
\(a_n^r\)Amount of resource \(r\) provided by one node of type \(n\)
\(D_s^r\)Total demand for resource \(r\) by service \(s\)

\(M_s\)Penalty cost per unit of unmet demand for service \(s\)
\(P_n\)Power draw (kW) of one node of type \(n\)
\(W_d\)Maximum power capacity (kW) at DC \(d\)
\(B\)Total monthly budget cap

Complete MIP Formulation

Objective — Minimize Total Cost + Penalties $$\min \underbrace{\sum_{n \in \mathcal{N}} \sum_{d \in \mathcal{D}} c_n \cdot x_{n,d}}_{\text{Node procurement cost}} \;+\; \underbrace{\sum_{s \in \mathcal{S}} M_s \cdot u_s}_{\text{Unmet demand penalty}}$$
Constraint 1 — Resource Sufficiency (for each resource, each DC) $$\sum_{n \in \mathcal{N}} a_n^r \cdot x_{n,d} \;\geq\; \sum_{s \in \mathcal{S}} D_s^r \cdot y_{s,d} \quad \forall \; r \in \mathcal{R}, \; d \in \mathcal{D}$$

Total resource provided by all nodes at DC \(d\) must cover the demand assigned there.

Constraint 2 — Demand Coverage (for each service) $$\sum_{d \in \mathcal{D}} y_{s,d} + u_s \geq 1 \quad \forall \; s \in \mathcal{S}$$

The fraction of service \(s\)'s demand served across all DCs, plus unmet fraction, must be at least 1 (100%). If \(u_s > 0\), some demand is unmet and incurs the penalty.

Constraint 3 — Power Capacity (for each DC) $$\sum_{n \in \mathcal{N}} P_n \cdot x_{n,d} \leq W_d \quad \forall \; d \in \mathcal{D}$$

Total power draw of all nodes at DC \(d\) must not exceed the facility's power envelope.

Constraint 4 — Budget $$\sum_{n \in \mathcal{N}} \sum_{d \in \mathcal{D}} c_n \cdot x_{n,d} \leq B$$

Total monthly infrastructure spend cannot exceed the budget.

Constraint 5 — Affinity (KV Cache pinned to DC-West) $$y_{s_3, d_2} = 0$$
Constraint 6 — Integrality & Non-Negativity $$x_{n,d} \in \mathbb{Z}_{\geq 0} \quad \forall \; n, d$$ $$y_{s,d} \geq 0, \quad u_s \geq 0 \quad \forall \; s, d$$

Interpreting the MIP Gap

LP relaxation bound: $319,580/month (the best possible if fractional servers were allowed).

Best integer solution: $322,160/month.

MIP Gap: \(\frac{322{,}160 - 319{,}580}{322{,}160} = 0.80\%\).

This means the "integrality penalty" — the extra cost of buying whole servers vs. the theoretical fractional optimum — is about $2,580/month (0.8%). This is very tight; the solution is essentially optimal for practical purposes.

Sensitivity Analysis

What happens if the budget is reduced to $300K?

The budget constraint becomes binding. The optimizer must shed some demand (\(u_s > 0\)). Given the penalty structure, it will first reduce allocation for the lowest-tier service. If ML Inference batch jobs have \(M_s = \$5{,}000\), the optimizer might leave 50 GPUs worth of inference demand unmet, saving $87,750 in GPU-Infer nodes at a penalty cost of $250,000 in unmet demand — which still exceeds savings. More likely, it switches to fewer GPU-Train nodes (expensive) and packs more onto GPU-Infer. The shadow price on the budget constraint tells you exactly how much one extra dollar of budget is worth.

What happens if GPU-Train node cost drops by 20%?

GPU-Train monthly cost drops from $6,200 to $4,960. The optimizer will likely substitute some GPU-Infer nodes with GPU-Train nodes, because each GPU-Train node provides 8 GPUs vs. 4 for GPU-Infer — the per-GPU cost of GPU-Train is now \(\$4{,}960/8 = \$620\) vs. \(\$1{,}350/4 = \$337.50\) for GPU-Infer. Wait — GPU-Infer is still cheaper per GPU, so the optimizer might not change much. However, GPU-Train also provides more CPU and RAM, so if those resources are binding, the optimizer might prefer GPU-Train for its "free" CPU/RAM. This illustrates why you need the solver — human intuition fails with multi-dimensional trade-offs.

Cost Coefficient Ranging: How stable is the Compute-L selection?

Cost coefficient ranging asks: "Over what range of \(c_{\text{Compute-L}}\) does the current basis remain optimal?" Suppose the range is $700–$1,100/month. Since the current cost is $870, there's a $230 upward margin before the optimizer would switch to an alternative (perhaps Memory-XL, which provides more RAM per server at slightly higher cost). This gives confidence the plan is robust to moderate vendor price negotiations or hardware cost fluctuations.

MIP vs. LP — The Integrality Story

The LP relaxation says "buy 42.3 Compute-L nodes." The MIP rounds this to 43 (ceiling) or finds a better combination by shifting resources to other node types. The 0.8% MIP gap represents the "cost of indivisibility." For strategic planning, you can often use the LP relaxation alone (faster, exact duals). For procurement, you need the MIP because purchase orders are for whole servers.

Bin Packing — Detailed Example

Given a fleet of servers (already procured), assign individual service instances to specific machines. This is the operational scheduling problem that runs daily or in real-time. The goal: pack workloads onto the fewest machines possible (or maximize utilization).

Scenario Setup

Problem Data — Single-Dimension (CPU) Example First

Available servers (bins): 5 identical Compute-L machines, each with \(C = 192\) CPU cores.

Service instances (items) to place:

Instance \(i\)ServiceCPU requirement \(w_i\)
1–20Photos API (20 replicas)24 cores each
21–30KV Cache (10 replicas)48 cores each
31–38ML Inference (8 replicas)16 cores each
39–42Batch Analytics (4 replicas)64 cores each

Total CPU demand: 20×24 + 10×48 + 8×16 + 4×64 = 480 + 480 + 128 + 256 = 1,344 cores.

Total capacity: 5 × 192 = 960 cores. Not enough! We need at least \(\lceil 1344/192 \rceil = 7\) servers. Assume we actually have 8 servers available (some buffer).

Complete Variable Dictionary

VariableDomainMeaning
\(x_{ij} \in \{0,1\}\)Binary1 if instance \(i\) is assigned to server \(j\); 0 otherwise
\(z_j \in \{0,1\}\)Binary1 if server \(j\) is used (has at least one instance); 0 if empty

Parameters:

ParameterMeaning
\(n = 42\)Total number of instances (items) to place
\(m = 8\)Total number of available servers (bins)
\(w_i\)CPU cores required by instance \(i\) (item size)
\(C = 192\)CPU core capacity of each server (bin capacity)

Complete Bin Packing Formulation

Objective — Minimize Number of Servers Used $$\min \sum_{j=1}^{m} z_j$$
Constraint 1 — Capacity (each server's load ≤ capacity) $$\sum_{i=1}^{n} w_i \cdot x_{ij} \leq C \cdot z_j \quad \forall \; j = 1, \ldots, m$$

If \(z_j = 0\) (server not used), the RHS is 0, forcing all \(x_{ij} = 0\) for that server. If \(z_j = 1\), total load ≤ \(C = 192\).

Constraint 2 — Assignment (each instance goes to exactly one server) $$\sum_{j=1}^{m} x_{ij} = 1 \quad \forall \; i = 1, \ldots, n$$

Every instance must be placed on exactly one server. No splitting.

Constraint 3 — Binary $$x_{ij} \in \{0,1\}, \quad z_j \in \{0,1\}$$
Optimal Packing for This Example
Server 1 (192 cores): [Photos×4 = 96] + [KV×2 = 96] = 192 ✓ FULL Server 2 (192 cores): [Photos×4 = 96] + [KV×2 = 96] = 192 ✓ FULL Server 3 (192 cores): [Photos×4 = 96] + [KV×2 = 96] = 192 ✓ FULL Server 4 (192 cores): [Photos×4 = 96] + [KV×2 = 96] = 192 ✓ FULL Server 5 (192 cores): [Photos×4 = 96] + [KV×2 = 96] = 192 ✓ FULL Server 6 (192 cores): [ML-Inf×8 = 128] + [Batch×1 = 64] = 192 ✓ FULL Server 7 (192 cores): [Batch×3 = 192] = 192 ✓ FULL Server 8 (192 cores): — not used — = 0 ✗ EMPTY ───────────────────── Servers used: 7 of 8. Total utilization: 1,344/1,344 = 100% on used servers.

The optimal solution uses 7 servers, matching the theoretical lower bound \(\lceil 1344/192 \rceil = 7\). Server 8 remains idle — it can be powered down to save OPEX.

Heuristics for Large-Scale Bin Packing

Exact bin packing (solving the BIP above) is NP-hard and impractical for 100K+ instances. Instead, production schedulers use polynomial-time heuristics:

HeuristicApproachApproximation RatioTime
First Fit Decreasing (FFD) Sort items by size (largest first). Place each item in the first bin that has enough room. Open a new bin if none fit. \(\leq \frac{11}{9} \text{OPT} + \frac{6}{9}\) O(n log n)
Best Fit Decreasing (BFD) Sort items largest first. Place each in the bin with the least remaining room that still fits. Minimizes wasted space per bin. Same as FFD asymptotically O(n log n)
First Fit (FF) Process items in arrival order (no sort). Place in first available bin. \(\leq \frac{17}{10} \text{OPT} + \frac{7}{10}\) O(n log n)
Next Fit (NF) Only try the current bin. If it doesn't fit, close it and open a new one. \(\leq 2 \cdot \text{OPT}\) O(n)
Why FFD Dominates in Practice

First Fit Decreasing is the workhorse of production schedulers (Kubernetes, Borg, etc.). By placing large items first, it avoids the "small items fill bins, then a large item can't fit anywhere" pathology. The \(11/9\) approximation means it uses at most ~22% more bins than optimal — and in practice, it's usually within 1–5% of optimal. It runs in O(n log n) time, making it suitable for real-time scheduling of millions of containers.

Multi-Dimensional (Vector) Bin Packing

Real capacity planning involves multiple resource dimensions simultaneously. An instance needs CPU and RAM and possibly GPU and network bandwidth. The formulation extends naturally:

Multi-Dimensional Bin Packing $$\min \sum_{j=1}^{m} z_j$$ $$\text{s.t.} \quad \sum_{i=1}^{n} w_i^r \cdot x_{ij} \leq C_j^r \cdot z_j \quad \forall \; j, \; \forall \; r \in \mathcal{R}$$ $$\sum_{j=1}^{m} x_{ij} = 1 \quad \forall \; i$$ $$x_{ij} \in \{0,1\}, \quad z_j \in \{0,1\}$$
New SymbolMeaning
\(w_i^r\)Requirement of instance \(i\) for resource \(r\) (e.g., \(w_5^{\text{ram}} = 64\) GB)
\(C_j^r\)Capacity of server \(j\) for resource \(r\) (servers may be heterogeneous)
\(\mathcal{R}\)Set of resource dimensions: {cpu, ram, gpu, storage, network}
Multi-Dimensional Complexity

Vector bin packing is much harder than 1-D. The FFD heuristic no longer has a bounded approximation ratio — in the worst case, it can use \(\Omega(d)\) times the optimal number of bins, where \(d\) is the number of dimensions. In practice, the dominant resource heuristic helps: sort items by their largest relative dimension (e.g., if an item uses 50% of a server's RAM but only 10% of CPU, RAM is dominant). Place items by dominant resource first, then fit smaller dimensions greedily.

So in 2-D packing, the number of bins needed is driven by the tightest dimension, and utilization on other dimensions suffers — this is the stranding problem from Section 4 of the Cost Modeling document.

Anti-Affinity and Spread Constraints in Bin Packing

Production schedulers add constraints beyond simple capacity:

  • Anti-affinity: No two replicas of the same service on the same server (for fault tolerance). Adds: \(x_{i_1, j} + x_{i_2, j} \leq 1\) for replicas \(i_1, i_2\) of the same service.
  • Spread: Replicas must be spread across racks/failure domains. Adds: at most \(k\) replicas per rack.
  • Affinity: Certain pairs of services should co-locate (low-latency communication). Adds: \(x_{i_1, j} = x_{i_2, j}\) for paired instances.
  • Preemption priority: High-priority services can evict lower-priority ones. Modeled with multiple rounds of packing or priority-ordered assignment.

Each constraint makes the bin packing harder to solve but more operationally realistic.

INTERVIEW PREP

Interview Q&A

Six practice questions covering LP, MIP, and bin packing in the context of iCloud capacity planning.

Q1: When would you use an LP instead of a MIP for capacity planning?

I'd use an LP in three situations. First, for strategic sizing at a 6–24 month horizon where the question is "how much total CPU, RAM, and GPU capacity do we need across regions?" — at this granularity, fractional values are meaningful because they represent large quantities where rounding error is negligible (if the LP says 10,437.3 cores, buying 10,438 is fine). Second, to obtain exact shadow prices: LP duality gives mathematically rigorous marginal resource values that tell us which constraints are binding and what each additional unit of capacity is worth. MIP doesn't provide exact duals. Third, as a warm start for the MIP: solving the LP relaxation first gives a lower bound and a good initial solution for branch-and-bound, often reducing MIP solve time by 50% or more.

I'd switch to MIP when the decisions are inherently discrete — buying whole servers, adding racks, or enabling/disabling data center pods — because the LP relaxation can be misleading when integer effects are large (e.g., buying 0.3 of a $300K GPU node is not meaningful).

Follow-up: Can you ever use the LP solution directly for procurement?
Yes, if the quantities are large enough that rounding has negligible cost impact. If the LP says "buy 847.2 Compute-L nodes" and you round to 848, the cost error is one server (~$870/month on a $700K total — 0.1%). But if the LP says "buy 2.7 GPU-Train nodes," rounding to 3 adds $6,200/month on a small base. The rule of thumb: if LP variables are consistently > 50 units, rounding is fine; below 10, you need the MIP for meaningful precision.
Q2: Explain shadow prices and how you'd use them to prioritize infrastructure investments.

Shadow prices (dual variables from the LP) quantify the marginal value of relaxing each constraint by one unit. In capacity planning terms, they answer: "if we had one more GPU / GB of RAM / MW of power, how much would the total system cost decrease?"

Here's how I'd use them. Suppose our LP gives these shadow prices: GPU at DC-West = $800/GPU-month, RAM at DC-West = $0.45/GB-month, CPU everywhere = $0/core-month, Power at DC-West = $75,000/MW-month. This tells me: (1) CPU is not a bottleneck — we have slack, so spending on compute nodes is wasteful. (2) RAM is moderately valuable — every GB freed up saves $0.45/month, so optimizing service memory footprints has measurable ROI. (3) GPUs are extremely scarce — at $800/GPU-month, any project that reduces GPU usage by even 10 GPUs saves $8K/month. (4) Power at DC-West is a hard ceiling — the $75K/MW shadow price means a 1 MW expansion justified by $75K/month in cost savings is equivalent to $900K/year. If the expansion costs less than that annualized, do it.

I'd present these shadow prices to the finance team as the basis for the next budget cycle: rank investment proposals by their shadow-price-implied ROI.

Q3: What is the MIP gap, and how do you decide when to stop the solver?

The MIP gap is \(\frac{z_{\text{best feasible}} - z_{\text{LP bound}}}{z_{\text{best feasible}}} \times 100\%\), where \(z_{\text{best feasible}}\) is the objective value of the best integer solution found so far and \(z_{\text{LP bound}}\) is the LP relaxation's optimal value (a lower bound for minimization). A gap of 0% means proven optimal; a gap of 2% means the solution might be up to 2% worse than optimal.

I'd set the termination criteria based on the decision stakes. For an annual $50M capacity plan, 1% gap = $500K uncertainty — worth waiting hours for a tighter solution. For a weekly rebalancing of a $200K/month cluster, 5% gap = $10K — not worth more than a few minutes of solver time. I also set a wall-clock time limit (e.g., 30 minutes) as a hard stop, because solver convergence often follows a pattern: rapid improvement in the first few minutes, then diminishing returns. If after 30 minutes the gap is 3% and hasn't improved in the last 10 minutes, the current incumbent is effectively as good as we'll get without structural reformulation.

Follow-up: What if the MIP gap is stuck at 15%?
A large gap usually means the LP relaxation bound is weak — the continuous relaxation is too loose an approximation of the integer problem. I'd try: (1) adding cutting planes (Gomory cuts, knapsack covers) to tighten the relaxation, (2) reformulating with better branching priorities (branch on the most fractional variables first), (3) providing a warm-start solution from a heuristic (e.g., FFD bin packing), (4) decomposing the problem (e.g., Benders decomposition by DC, solving each DC sub-problem separately then coordinating). If none of these work within the time budget, I'd fall back to the heuristic solution and accept the gap, documenting the uncertainty for the planning team.
Q4: How does bin packing relate to the MIP procurement model in a capacity planning pipeline?

They operate at different time scales and granularities in a two-stage pipeline. The MIP runs quarterly or monthly to decide the fleet composition: how many of each server type to buy and deploy at each data center. Its decision variables are integer counts of servers. The bin packing runs continuously (real-time or hourly) to decide the workload placement: which specific service instances go on which specific machines. Its variables are binary assignments.

The connection is that the MIP's output (the purchased fleet) becomes the bin packing's input (available bins). If the MIP solution has high resource stranding — say, it bought enough RAM but has 30% excess CPU — the bin packing will confirm this by showing low CPU utilization across the fleet. This utilization feedback is then used in the next MIP solve cycle to adjust the node mix (perhaps fewer Compute-L nodes, more Memory-XL).

There's also a feasibility check: the MIP assumes workloads can be packed onto the purchased servers. But the MIP doesn't model instance-level placement constraints (anti-affinity, rack spread). It's possible (though rare) for a MIP-optimal fleet to have packing issues if, say, anti-affinity rules prevent placing all replicas even though aggregate capacity suffices. Running a bin packing simulation post-MIP catches this before hardware is ordered.

Q5: Walk me through how penalty variables enable service tiering in the optimization.

The idea is to convert hard demand constraints into soft constraints using slack variables with different costs. For each service \(s\), instead of the hard constraint \(\sum_d y_{s,d} \geq 1\) (100% of demand must be met), we write \(\sum_d y_{s,d} + u_s \geq 1\) where \(u_s \geq 0\) is the unmet fraction. We then add \(M_s \cdot u_s\) to the objective, where \(M_s\) is the penalty cost per unit of unmet demand.

For iCloud, I'd tier services like this. Tier-0 (authentication, iCloud Drive sync): \(M = \$500{,}000\)/unit — effectively infinite; the optimizer will buy any amount of hardware before shedding this. Tier-1 (Photos, Mail): \(M = \$50{,}000\)/unit — very expensive to leave unmet, but the optimizer can shed small amounts under extreme budget pressure. Tier-2 (analytics, offline ML training): \(M = \$2{,}000\)/unit — the optimizer will shed these first when budget is tight.

Under a budget crunch, the optimizer gracefully degrades: it meets all Tier-0 demand, meets most Tier-1, and sheds Tier-2 until the budget constraint is satisfied. The penalty costs encode the business priority ordering without requiring manual intervention. And the shadow price on the budget constraint tells you exactly how much one more dollar of budget is worth in terms of prevented demand shedding.

Follow-up: How do you set the penalty values?
In practice, penalty values come from two sources. First, business impact estimates: if 1% of iCloud Auth is unavailable for a month, what is the revenue/brand impact? Convert that to $/unit-month. Second, iterative tuning: set initial values, run the optimizer, check if the shedding behavior matches business expectations (e.g., "analytics should be shed before photos"), adjust penalties until the ordering is correct. The absolute values matter less than the relative ordering — what matters is \(M_{\text{Tier-0}} \gg M_{\text{Tier-1}} \gg M_{\text{Tier-2}}\). A common heuristic is 10× between tiers.
Q6: Describe the First Fit Decreasing heuristic and why it works well for cluster scheduling.

First Fit Decreasing (FFD) is a greedy bin packing heuristic with two steps. First, sort all items (service instances) by size in decreasing order. Second, for each item, scan the bins (servers) in order and place the item in the first bin with enough remaining capacity. If no existing bin fits, open a new one.

It works well for several reasons. The decreasing sort ensures large items are placed first, when bins are empty and have the most room. This prevents the pathology where small items fill bins to near-capacity, and then a large item arrives that doesn't fit anywhere — forcing an unnecessary new bin. The theoretical guarantee is \(\text{FFD}(I) \leq \frac{11}{9} \text{OPT}(I) + \frac{6}{9}\), meaning it never uses more than about 22% extra bins. In practice, it's almost always within 1–5% of optimal for real workload distributions.

For cluster scheduling at iCloud scale (millions of containers), FFD's O(n log n) time complexity is critical. An exact MIP solution would take hours for millions of instances; FFD solves it in seconds. The trade-off (potentially a few extra servers) is acceptable because servers are already purchased — the packing decision is about utilization, not procurement cost. And in Kubernetes-style schedulers, the "bins" are periodically rebalanced, so any FFD sub-optimality is corrected over time.

Follow-up: How does this extend to multiple dimensions?
In multi-dimensional bin packing (CPU + RAM + GPU), FFD's guarantee no longer holds because you can't sort items by a single "size." The common approach is Dominant Resource FFD: for each item, compute its largest relative consumption across dimensions (e.g., an item using 50% of a server's RAM and 10% of CPU has RAM as its dominant resource). Sort items by dominant resource consumption descending, then apply FFD. Another approach is scoring: define a scalar score like \(\text{score}_i = \max_r(w_i^r / C^r)\) and sort by that. Google's Borg scheduler uses a variant of this with multi-factor scoring that includes memory, CPU, and anti-affinity penalties.
